# 테스터(모리 TESTER) 설치 — 단일프로젝트 (윈도우-first)

> 이 문서는 UnSkein 플러그인과 함께 배포되는 일반 설치 가이드다. (v1.28.0)
>
> 윈도우 호스트 위에 **모리 TESTER** 를 세우는 절차. TESTER = UNSKEIN 큐에서 **`test`(화면검증) 작업**을 선점하고, **화면·런타임을 실제 브라우저(Chrome CDP)로 검증**한다 — 콘솔 에러·네트워크 실패 수집 + 실제 렌더 확인 + 화면 캡처. 검증에 쓴 시나리오 스크립트·스크린샷은 남겨 사용자 매뉴얼 작성에 재사용한다.
>
> 🔑 **TESTER = 윈도우 네이티브(WSL 아님).** EXECUTOR(`익스큐터설치_멀티프로젝트.md`)는 헤드리스 WSL이지만, TESTER는 **브라우저(Chrome CDP 9222)가 필수**라 윈도우에서 직접 돈다 — 역할 분리 중 유일한 "환경 필연" 분리다. 헤드리스 WSL은 브라우저를 못 띄우고, WSL 안에서 `127.0.0.1:9222` 를 열어도 그것은 WSL 루프백이지 윈도우 Chrome이 아니다(§2·§8).
>
> ⚠️ **현재 상태:** **서버 배선은 완비**(kind=tester claim 게이트 · `tested_url` 전달 · `payload['test']` 결과 슬롯 · CDP 의뢰 자동등록). 클라이언트는 `unskein-test` 스킬이 **수동 CDP 툴킷 + 자율 tick(`queue.js`: 선점→검증→보고)** 을 제공한다. 다만 상시 데몬(EXECUTOR의 `run_loop` 상당)은 없어 연속 운용은 세션 폴링 또는 주기 재구동으로 하는 **반자동**이다(§7). 셋업은 `unskein-setup` §T(TESTER 프로비저닝 — 프로젝트별 번들)로 하거나, 단일 배치면 `tester.env` 를 수동 작성한다(§4·§4.1).
>
> 출처: 설계 결정 ADR-0014(테스터 배선)·ADR-0019(CDP 검증 의뢰), 플러그인 `skills/unskein-test/SKILL.md`. 짝 문서: `익스큐터설치_멀티프로젝트.md`(EXECUTOR).

## 0. TESTER vs EXECUTOR — 무엇이 다른가

| | **EXECUTOR**(익스큐터설치) | **TESTER**(이 문서) |
|---|---|---|
| 환경 | WSL distro, **헤드리스** | **윈도우 네이티브** (Chrome CDP 9222) |
| 토큰 | kind=**mori** | kind=**tester** (같은 `X-Mori-Token`→`/api/mori/*`, claim만 kind로 갈림) |
| claim 대상 | `plan`/`exec`/`inspect` (+answered) | **`test`** (+ test에서 막힌 answered) |
| 하는 일 | 다오 구동 → 코드 개발 → PR | **화면검증(CDP)** + 검증 산출물(시나리오·캡처) 축적 |
| 런타임 | 프로젝트 스택(node/py) | **윈도우 Chrome + 윈도우 Node + Playwright(npm)** |
| 플러그인 | unskein-plugin(orchestrator+dao-skills) | **unskein-plugin**(unskein-test) |
| 실행 | `/unskein:watch`(run_loop 자율) | **반자동**(`queue.js` tick + 필요 시 주기 재구동 — §7) |

핵심: **TESTER 설치 = "윈도우에 Chrome/Node/Playwright + unskein 플러그인 + kind=tester 토큰 + 그 프로젝트 watch scope"를 갖춘 세션.**

> **달리는 예시**: 비즈니스 `ACME` / 프로젝트 `WebApp` 한 개를 담당하는 윈도우 TESTER 하나. 이 문서 전체에서 이 예시 값을 쓴다 — 자기 비즈니스/프로젝트 이름으로 바꿔 읽으면 된다.

## 1. 동작 원리 — 윈도우 한 대, 화면검증

```
  [윈도우 TESTER]
     Claude Code 세션 (kind=tester 토큰)
        │  UNSKEIN 큐 폴링(반자동) → claim(status=test)
        │      → unskein-test(CDP) 로 배포된 tested_url 검증
        │      → report(PASS=inspect/done · FAIL=plan)
        │
     Chrome (CDP 9222, 전용 프로필 .cdp-chrome)  ← 윈도우 네이티브
```

- **화면검증**은 UNSKEIN 큐의 `test` 작업 + CDP 의뢰 자동등록(화면검증이 필요한 작업이 완료될 때 검증 카드가 자동 생성)으로 들어온다. TESTER는 배포된 테스트서버/앱 URL(`tested_url`)을 CDP로 열어 콘솔·네트워크·렌더를 확인한다.
- 파이프라인 위치: `plan(구현+자체검증, EXECUTOR)` → **`test`(화면검증, TESTER)** → `inspect(마감, EXECUTOR)` → `done`. 구현자와 검증자가 프로세스 경계(WSL ↔ 윈도우)로 갈려 "구현 ≠ 검증"이 실현된다.
- 프로젝트에 따라 같은 윈도우 머신이 화면검증 외의 다른 도구를 겸할 수 있다(별도 플러그인) — 이 문서는 화면검증(CDP)만 다룬다.

## 2. 사전 준비 — 윈도우에

> **EXECUTOR와 정반대: WSL이 아니라 윈도우에** 런타임을 갖춘다. `unskein-test`의 `.ps1`은 PowerShell, `remote.js`는 윈도우 Node로 돌아야 `127.0.0.1:9222`가 윈도우 Chrome을 가리킨다(WSL Node면 루프백이 어긋난다 — SKILL.md §6).

| 도구 | 왜 | 확인 |
|------|----|----|
| **시스템 Chrome** | CDP 9222 검증 대상 브라우저 | `start.ps1`이 표준 경로 자동 탐색(`C:\Program Files\Google\Chrome\...`) |
| **윈도우 Node** | `remote.js`·`queue.js` 구동 | `node --version` (윈도우 PowerShell/CMD에서) |
| **Playwright(npm)** | CDP 원격조작(`connectOverCDP`) | 스킬 폴더(또는 상위)에서 `npm i -D playwright` 한 번. **`npx playwright install` 불필요**(번들 브라우저 안 씀 — 시스템 Chrome에 붙는다) |
| **Claude Code** | TESTER 세션 호스트 | 윈도우 Claude Code(Desktop 또는 CLI) |
| `git`·`gh` | (선택) 검증 산출물(캡처·시나리오)을 repo에 커밋할 때 | 대상 repo 권한 계정 — §6 |

- ⚠️ **CDP 도달은 윈도우 실환경에서만 확정**. WSL 자체 확인으로는 9222 도달을 보장 못 한다(SKILL.md §6).

## 4. 인증 — kind=tester 토큰 (`~/.unskein/tester.env`)

EXECUTOR의 executor.env와 같은 방식이되 **토큰 kind만 tester**:

1. 웹 로그인(예: `https://unskein.mupai.studio`) → 토큰 발급 **kind=`tester`**(⚠️ mori·planner 아님). 웹 드롭다운에 tester가 안 보이면 API로: `POST /api/me/mori-tokens {"name":"webapp-tester","kind":"tester"}` (사람 JWT 인증 — 백엔드가 tester kind 를 허용한다).
2. `~/.unskein/tester.env`(chmod 600):
   ```shell
   export UNSKEIN_API_BASE=https://unskein.mupai.studio
   export UNSKEIN_MORI_TOKEN=<kind=tester 토큰>       # X-Mori-Token 자리(이름은 MORI지만 tester 토큰)
   export UNSKEIN_WATCH_BUSINESS="ACME"
   export UNSKEIN_WATCH_PROJECT="WebApp"
   ```
   > PowerShell 세션이면 `$env:UNSKEIN_API_BASE="..."` / `$env:UNSKEIN_MORI_TOKEN="..."` 식으로 넣는다(값은 화면에 출력하지 않는다 — 비밀 무잔존).
3. 확인:
   ```shell
   curl -s -o /dev/null -w '%{http_code}\n' -H "X-Mori-Token: $UNSKEIN_MORI_TOKEN" "$UNSKEIN_API_BASE/api/mori/scope"   # 200
   ```
   `200` = tester 토큰 정상. `401` = kind 오배정/서버-토큰 짝 어긋남. `node queue.js scope` 로도 담당 가능 사이트를 확인할 수 있다.

### 4.1 디렉토리 구성 — 단일 vs 프로젝트별 번들

테스터는 검증 대상이 코드가 아니라 **배포된 화면(URL)** 이라 로컬에 프로젝트 폴더가 없다. 그래서 상태 앵커가 프로젝트 폴더가 아니라 **윈도우 홈 아래**에 온다. 두 배치 중 하나를 고른다:

**단일 배치** (프로젝트 하나 / 여러 사이트를 한 로그인·한 토큰으로 순차 담당 — 이 문서 기본):

```
%USERPROFILE%\.unskein\
└── tester.env                 ← 서버·tester 토큰·watch 범위 (§4)
%USERPROFILE%\.cdp-chrome\     ← CDP 전용 크롬 프로필(일반 크롬과 분리)
```

**프로젝트별 번들** (여러 프로젝트 병렬·사이트별 로그인 분리 — `unskein-setup` §T 가 생성):

```
%USERPROFILE%\.unskein\
└── <business>__<project>\     ← 프로젝트 번들(앵커) — 프로젝트마다 반복
    ├── tester.ps1             ← config: 토큰·API·watch 범위·CDP 포트
    └── cases\<task_id>\       ← 검증 산출물(시나리오 스크립트·스크린샷)
%USERPROFILE%\.cdp-chrome-<n>\ ← 포트↔프로필 1:1 CDP 프로필(§7.2 — 인증 경계)
```

실행기·플래너와 같은 "상태는 프로젝트별 격리, 코드(플러그인)는 전역 1벌" 사상이고, 루트 위치만 다르다. **구성은 셋업의 산출물** — 사람이 미리 만들지 않는다.

## 5. watch scope — 이 프로젝트로 고정

TESTER는 kind 게이트로 어차피 `test` 스테이지만 집으므로, 여러 사이트를 담당해도 claim이 test로 좁혀진다. 특정 사이트만 담당하려면 claim 때 `--business`/`--project` 로 좁힌다(§7.1 — 예: `queue.js claim --business="ACME" --project="WebApp"`). 검증 대상 **`tested_url`** 은 `project.settings` 또는 task `payload`에서 claim 응답에 실려 온다.

## 6. git 자격증명 (선택 — 산출물을 커밋할 때만)

화면검증 자체에는 repo 자격증명이 불필요하다(원격 URL만 검증). 검증 산출물(스크린샷·시나리오 스크립트)을 프로젝트 repo(예: `test-cases/`)에 커밋해 매뉴얼 작성에 재사용하려면 EXECUTOR와 동일 방식 — HTTPS 토큰은 tester.env, SSH 키는 `~/.unskein/creds/` 파일(`익스큐터설치_멀티프로젝트.md` 참조).

## 7. 실행 — 반자동 운용

**상시 데몬(EXECUTOR의 `run_loop` 상당)은 없다.** 한 tick(1건 처리)을 `queue.js` 로 돌리고, 연속 감시는 세션 폴링 또는 주기 재구동으로 한다.

### 7.1 한 tick — claim → 검증 → report

윈도우 Claude Code 세션에서 (`tester.env` 로드 후):

1. **선점**: `node ${CLAUDE_PLUGIN_ROOT}/skills/unskein-test/scripts/queue.js claim --business="ACME" --project="WebApp"` → `{claimed, task}`. `claimed:false` 면 조용히 종료(대기 작업 없음). task에는 `tested_url`, 수용 기준(`title`/`description`/`plan_doc`), `subtree`가 실려 온다.
2. **lease 유지**: 검증이 길면 주기적으로 `node queue.js heartbeat <id>`(60초 안팎으로 갱신).
3. **검증**: `unskein-test`로 CDP 기동·검증
   ```shell
   powershell -ExecutionPolicy Bypass -File ${CLAUDE_PLUGIN_ROOT}/skills/unskein-test/scripts/start.ps1 -Url <tested_url>
   node ${CLAUDE_PLUGIN_ROOT}/skills/unskein-test/scripts/remote.js collect 5000
   node ${CLAUDE_PLUGIN_ROOT}/skills/unskein-test/scripts/remote.js shot <name>
   ```
   필요하면 `navigate`/`click`/`type`/`wait`/`attrs` 로 수용 기준의 시나리오를 수행한다(상세는 SKILL.md §4). 이번 검증에 쓴 시나리오와 캡처는 작업별 폴더에 남긴다(매뉴얼 재사용).
4. **report**: PASS면 `node queue.js report <id> --status=inspect --summary="..." --payload=<payload.json>` — 파이프라인 다음 단계(inspect)로 넘어간다. CDP 의뢰 카드면 서버가 `done`으로 remap 한다(ADR-0019). FAIL이면 `--status=plan`(구현자 EXECUTOR가 다시 집는다) + 실패 근거를 `payload.findings` 에 담는다. 사양이냐 버그냐 판단이 필요하면 `node queue.js question <id> --text="..."` 로 사람에게 묻고 tick을 종료한다.
5. **정리**: `stop.ps1` 로 CDP Chrome을 정리한다(병렬 세션이면 `-Port <n>` 로 자기 세션만).

### 7.2 포트↔프로필 페어링 — 병렬 세션·인증 격리

**인증(쿠키·localStorage·JWT)의 경계는 탭이 아니라 프로필**이다. `start.ps1` 이 포트와 프로필을 1:1로 묶는다:

| 기동 | 프로필 | 용도 |
|------|--------|------|
| `start.ps1` (기본 9222) | `.cdp-chrome` | 단일 세션 |
| `start.ps1 -Port 9223` | `.cdp-chrome-9223` | 병렬 세션 — 9222와 로그인 완전 분리 |
| `start.ps1 -Port <n> -Profile <name>` | `.cdp-chrome-<name>` | 이름 있는 프로필(예: 역할 계정 병렬 검증) |

- 병렬 TESTER는 **자기 포트로**: `start.ps1 -Port <n>` + `$env:CDP_PORT=<n>`(또는 remote.js 명령마다 `--port=<n>`). 포트를 안 나누면 두 루프가 같은 Chrome(같은 로그인)에 붙어 인증이 섞인다.
- 페어링 충돌은 조용히 넘어가지 않고 거부된다 — 같은 프로필·다른 포트(Chrome이 기존 인스턴스에 위임해 새 포트가 영영 안 뜨는 함정)도, 같은 포트·다른 프로필(로그인 섞임)도 명확한 에러. `-Force` 는 그 페어링을 점유한 Chrome만 종료하고 재확보한다(무관한 세션·일반 Chrome 불가침).
- `stop.ps1 -Port <n>` = 그 세션만 정리(그 포트 + 짝 프로필). 옵션 없이 `stop.ps1` = 모든 CDP Chrome 정리(일반 Chrome 보호).

### 7.3 연속 운용

한 tick은 1건만 처리한다. 연속 감시는 Claude 세션이 폴링하거나 주기 재구동(예: 5분 간격 cron)으로 tick을 반복한다. 중복 선점은 서버가 lease(heartbeat 신선도)로 막으므로 여러 TESTER를 동시에 돌려도 **큐 측은** 안전하다 — 단 **CDP 측은 세션별 포트 분리가 전제**다(§7.2).

## 8. 막히면

- **CDP 9222 안 붙음** → WSL Node로 `remote.js`를 돌리지 않았는지(윈도우 Node여야 — WSL 루프백 ≠ 윈도우 9222). `start.ps1` READY 출력 확인. Chrome 설치 경로 확인.
- **포트가 영영 안 뜸 / 로그인이 섞임** → 포트↔프로필 페어링 위반(§7.2). `stop.ps1 -Port <n>` 후 재기동, 또는 `-Force` 로 그 페어링만 재확보.
- **claim 401** → 토큰 kind가 tester인지, API-토큰 짝. `curl .../api/mori/scope` 로 확인(§4).
- **claim 빈 응답(`claimed:false`)** → 큐에 `status=test` 작업 없음. CDP 의뢰 카드는 화면검증이 필요한 작업이 done 될 때 자동 생성되거나, 완료 카드의 버튼으로 사람이 만든다.
- 그 밖의 진단·복구는 `unskein-doctor` 스킬.
