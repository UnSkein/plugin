# 플래너(모리 PLANNER) 설치

> 이 문서는 UnSkein 플러그인과 함께 배포되는 일반 설치 가이드다. (v1.29.0)
>
> **처음이라 역할부터 모호하면**: 이 문서를 다 읽기 전에 `unskein-setup` 부터 실행해도 된다 — 토큰만 있으면 서버가 토큰 종류(kind)로 역할을 판별해 알맞은 절차로 분기한다(setup R0, ADR-0027 · 서버에 `/api/whoami` 필요). 토큰 값은 대화·셸 인자에 넣지 말고 `.unskein/setup.env` 에 편집기로 넣는다.

> 개발자/운영자 컴퓨터에 **모리 PLANNER** 를 세우는 절차 — **프로젝트별로 격리**해 한 머신으로 여러 프로젝트를 다루는 표준 방식(설계 결정 ADR-0021·0022). PLANNER = 대상 프로젝트의 **코드베이스를 보유**한 채 개발 의뢰를 받아 스코프·플랜을 내리고 **UNSKEIN(PM SaaS) 작업 큐에 타스크로 등록**하는 운영 세션(개발자가 쓰는 Claude Code). **직접 개발은 안 한다**(실행 측 EXECUTOR→다오 몫).
>
> 🔑 **기획자는 한 번에 한 프로젝트만 다루지 않는다.** 그래서 **프로젝트마다 격리된 `planner.env`**(그 프로젝트의 서버·토큰·git 키·비즈니스/프로젝트 이름)를 두고, 그 프로젝트 작업폴더에서 `source … && claude` 하면 그 서버·토큰·코드베이스로 붙는다 — 프로젝트끼리 안 섞인다. **전역 한 벌 방식은 폐기됐다**(§3.4). `UNSKEIN_HOME` 은 필수다.
>
> 🔑 **코드베이스는 이제 사람이 clone/pull 하지 않는다**(ADR-0022): 스콥을 시작하면 `unskein-scope`(§5) 가 코드베이스를 스스로 보장한다 — 없으면 clone, 있으면 `git pull --ff-only`. 최초 clone 도 스콥이 한다.
>
> 🔑 **인증 해석 = source 우선·cwd 폴백**(실행기와 같은 원칙, ADR-0021): 스킬(scope·wbs·drawio·doctor)이 서버 호출 전 `planner.env` 를 로드한다 — ① 셸 env → ② `$UNSKEIN_HOME/planner.env` → ③ cwd 에서 위로 첫 `.unskein/planner.env` → ④ 전역 `~/.unskein/planner.env`(하위호환).
>
> ⚠️ **용어 콜아웃(이 문서의 핵심)**: **UNSKEIN(대문자) = 등록하는 PM SaaS 플랫폼**(작업 큐를 가진 서버) ≠ **대상 프로젝트**(스코프할 코드베이스). 둘은 다르다. 이 문서의 달리는 예시는 비즈니스 `ACME` / 프로젝트 `WebApp` 으로 잡는다 — 실제 값은 자기 서버 표기에 맞게 바꿔 읽는다.
>
> ⚠️ EXECUTOR(다오 구동 클라이언트) 설치는 짝 문서 → 같은 폴더의 `익스큐터설치_멀티프로젝트.md`. 한 머신이 둘 다 겸할 수 있지만 역할·셋업이 다르다.

## 0. PLANNER vs EXECUTOR — 무엇을 세우는가

| | **PLANNER** (이 문서) | **EXECUTOR** (`익스큐터설치_멀티프로젝트.md`) |
|---|---|---|
| 정체 | 개발자의 Claude Code / 운영 세션 | `run_once`/`run_loop` 폴링 실행기(WSL distro) |
| 코드베이스 | **보유**(scope 가 자동 clone/pull 로 보장·최신화) | 없음 — 다오가 작업마다 클론 |
| 하는 일 | 의뢰 → 스코프·플랜 → **타스크 등록** | 큐 claim → 다오 WSL 구동 → 개발 → PR |
| 상태 | `planner.env`(서버 토큰·프로젝트 이름·git 키) + `creds/`(SSH 키) — `work`·run_loop 없음 | `executor.env` + creds + work |
| 쓰는 것 | 운영자 스킬 `unskein-scope`·`unskein-wbs`·`unskein-drawio`·`unskein-session`·`unskein-lock` 등 | `/unskein:run`·`/unskein:watch` + 다오 스킬 6개 |
| 서버 접근 | **planner 토큰**(kind=planner, `X-Planner-Token`) | **모리 토큰**(kind=mori, `X-Mori-Token`) |
| 개발 | **안 함** | 함(다오가) |

핵심: **PLANNER 설치 = "프로젝트마다 격리된 `.unskein`(`planner.env`=서버 토큰·프로젝트 이름·git 키 + `creds/`=SSH 키) 을 갖추고, 그 작업폴더에서 claude 를 띄우면 그 프로젝트로 붙게 하는 것".** 코드베이스 자체는 스콥이 자동으로 clone/pull 하므로 미리 받아 둘 필요가 없다(§5). 실행기와 달리 `work`·다오·run_loop 는 없다(사람이 대화하는 세션).

> **EXECUTOR 를 겸하는 머신**이면 추가로 `UNSKEIN_MORI_TOKEN`(kind=mori) 을 `unskein-setup` 으로 둔다 — 그건 claim 용이고 PLANNER 등록(planner 토큰)과는 별개다. 겸용 프로젝트는 `<프로젝트>/.unskein` 하나에 `executor.env`·`planner.env` 를 공존시킨다(ADR-0021).

## 1. 사전 준비

> 아래 표는 **확인용 체크리스트**다. 아무것도 없는 신규 머신은 먼저 **§1-A 설치 순서**를 따른다.

| 도구 | 왜 | 확인 |
|------|----|----|
| `claude` (Claude Code) | PLANNER 세션 + 플러그인(운영자 스킬) 호스트 | `claude --version` |
| `git` | 코드베이스 clone·pull·문서 PR | `git --version` |
| `gh` (인증됨) | 스콥 문서·스펙을 feature 브랜치로 PR + HTTPS repo 클론 자격 | `gh auth status` |
| `jq` | scope 가 서버 목록에서 프로젝트 이름 매칭·`repo_url` 추출(§5) | `jq --version` |
| `python3` | 스킬 보조(understand 그래프 머지 등) | `python3 --version` |
| `node`≥22·`pnpm`≥10 | understand-anything(코드 이해 그래프) 런타임 | `node --version` · `pnpm --version` |

- `gh` 인증: `gh auth login`(대상 repo 에 push/PR 권한 있는 계정, GitHub.com·HTTPS) → `gh auth setup-git`.
- PLANNER 는 WSL 안/밖 어디서든 된다(코드베이스만 보유). EXECUTOR 를 겸하면 그쪽 상주 규칙상 WSL distro 안 상주가 필요하다.

### 1-A. 신규 머신 — 처음부터 설치 순서 (Linux/WSL)

아무것도 없는 플래너 머신을 세우는 순서. **Claude Code 설치가 맨 앞**이다 — PLANNER 정체가 곧 "코드베이스를 보유한 Claude Code 세션"이라 `claude` 없이는 시작 자체가 안 된다.

**① Claude Code(`claude`) 설치 + PATH 등록 + 로그인**
```shell
curl -fsSL https://claude.ai/install.sh | bash      # claude 설치 (기본 ~/.local/bin)

# Linux: PATH 에 ~/.local/bin 등록 (설치 후 claude 명령이 안 잡히면 필수)
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc && source ~/.bashrc

claude --version                                    # 설치 확인
claude                                              # 첫 실행 → Anthropic 계정 로그인
```
> zsh 를 쓰면 `~/.bashrc` 대신 `~/.zshrc` 에 같은 줄을 넣는다. PATH 등록은 **한 번**이면 되고, 이미 잡혀 있으면(`command -v claude` 로 확인) 생략.

**② 베이스 런타임 — Claude Code 에게 맡긴다**
`claude` 가 설치되면 Bash 접근이 있으니, git·jq·python3·node≥22·corepack/pnpm·gh 설치는 **직접 안 하고 아래 프롬프트를 붙여넣어** Claude 에게 시킨다(OS·패키지 매니저 자동 판별). 인증만 사람 몫이다.

```text
이 머신을 unskein 플래너(모리 PLANNER)로 세우려고 한다.
먼저 `uname -a` 와 배포판을 확인하고, 이 OS 에 맞는 패키지 매니저로 아래 런타임을 설치·검증해줘.
이미 설치돼 있으면 건드리지 말고 버전만 확인해.

- git
- jq
- python3
- node ≥ v22   (없거나 낮으면 nvm 또는 배포판 공식 방법으로 v22 설치)
- corepack(node 동봉) → `corepack enable pnpm` 로 pnpm(≥10) 조달   # understand-anything 용(안 쓰면 생략 가능)
- gh (GitHub CLI) — 설치까지만. 인증(gh auth login)은 내가 직접 한다.

각 도구는 설치 후 버전을 출력해 확인하고, 실패하거나 sudo·전역 설정 변경이 필요하면
임의로 진행하지 말고 멈춰서 물어봐.
```

**③ 인증 — 사람이 직접 (설치와 별개)**
```shell
claude                                              # 아직 로그인 안 했으면: Anthropic 계정 로그인
gh auth login   &&   gh auth setup-git              # 대상 repo 권한 계정, GitHub.com·HTTPS
```
> 최종 확인은 §1 체크리스트 표대로. **WSL 일 때만** `wslu`(`sudo apt install wslu`) 추가 권장 — `claude`/`gh` 로그인·대시보드 URL 을 윈도우 브라우저로 여는 데 쓴다.

이후 **§2 플러그인 → §3 planner.env 배치·source → §4 git 키 → §5 자동 clone/pull** 순으로 이어간다.

## 2. 플러그인 설치 (운영자 스킬, 전역 1벌)

운영자 스킬은 `unskein` 플러그인에 들어 있다(public `UnSkein/plugin` — 마켓플레이스 최신 설치).

```shell
claude plugin marketplace add UnSkein/plugin
claude plugin install unskein@unskein-marketplace
claude plugin list        # unskein 버전 확인
```

- **버전 요건**: 프로젝트별 `planner.env` 격리·source 우선·cwd 폴백은 **v1.20.0+**(ADR-0021), **자동 clone/pull + `planner.env` 3변수**(`UNSKEIN_BUSINESS`·`UNSKEIN_PROJECT`·`UNSKEIN_SSH_KEY`)는 **v1.21.0+**(ADR-0022). 이 문서 절차는 **v1.21.0 이상**을 전제한다.
- 코드(스킬)는 **전역 1벌 공유** — 여러 프로젝트가 같은 스킬을 쓴다. 프로젝트별로 갈리는 건 상태(`planner.env`)뿐이다.
- **버전이 낡게 잡히면**: `claude plugin marketplace update unskein-marketplace` → `claude plugin update unskein@unskein-marketplace` → **claude 재시작**. 1.21.0 미만이면 자동 clone/pull 이 없어 코드베이스를 사람이 직접 받아야 한다.
- 설치 확인: `/unskein:status`(읽기 전용) 또는 `claude` 에서 `unskein-scope` 트리거가 잡히는지.

> **plugin repo 소스 클론은 PLANNER 필수가 아니다.** 운영자 스킬은 마켓플레이스 설치로 충분하다.

## 3. planner.env 배치 + source

> PLANNER 는 실행기의 `unskein-setup` 같은 셋업 스킬을 안 쓴다(상태가 가벼워 — `planner.env`(서버·프로젝트) + `creds/`(git 키)뿐, 다오·work 없음 — 수동이 더 단순). 아래로 `planner.env` 를 두고, §4 로 git 키를 둔다.

### 3.1 배치 구조 (프로젝트별 격리)

프로젝트마다 **작업폴더** 하나를 정하고, 그 안에 `.unskein/`(상태 루트)를 둔다. 코드베이스는 그 `.unskein` 의 **부모**(= 작업폴더) 아래로 스콥이 자동 clone 한다. 예:

```
~/work/webapp/                   ← 프로젝트 작업폴더 (UNSKEIN_HOME 의 부모)
├── .unskein/
│   ├── planner.env              ← 서버 토큰·프로젝트 이름·git 키 (chmod 600)
│   └── creds/
│       └── id_ed25519           ← SSH 키 (chmod 600) — §4
└── WebApp/                      ← 코드베이스: scope 가 자동 clone (사람이 안 받는다)
```

- 🔑 **`.unskein` 은 코드베이스 repo 밖**에 둔다(작업폴더 바로 아래, 코드베이스는 그 옆). 토큰·git 키가 고객 repo 커밋·`git clean` 에 안 섞이게 — 실행기가 clone 을 `work/` 밑에 두는 것과 같은 이유(ADR-0021).
- 역할별 비교: 실행기는 `.unskein/` 밑에 `work/`(다오 작업 루트)가 더 있고(`익스큐터설치_멀티프로젝트.md` §4.0), 테스터는 윈도우 홈 아래 프로젝트 번들이다(`테스터설치_단일프로젝트.md` §4.1). 셋은 모두 "상태는 프로젝트별 격리, 코드(플러그인)는 전역 1벌" — 디렉토리 구성은 셋업의 산출물이다.

### 3.2 planner 토큰 발급 + planner.env 작성

1. **웹에서 planner 토큰 발급**: 서버(예: `https://unskein.mupai.studio`) 로그인 → 설정에서 발급 — **kind 반드시 `planner`**(⚠️ mori 아님). **딱 한 번 보임 — 즉시 복사.** 토큰 이름은 프로젝트 전용으로(예: `planner-webapp`).
2. **템플릿 복사 후 편집기로 채운다**(echo/명령줄 금지 — 셸 기록에 남음):
```shell
TPL=$(find ~/.claude/plugins/cache/unskein-marketplace -name planner.env.sample | sort -V | tail -1)
PROJ=~/work/webapp                                     # ← 이 프로젝트 작업폴더(이름 자유)
mkdir -p "$PROJ/.unskein/creds"
cp "$TPL" "$PROJ/.unskein/planner.env" && chmod 600 "$PROJ/.unskein/planner.env"
```
3. **`planner.env` 값 (편집기로 — 각 값 `export`)**:
```shell
export UNSKEIN_HOME=/home/<user>/work/webapp/.unskein     # 절대경로, 파일 맨 위. ~·상대경로 금지
export UNSKEIN_API=https://unskein.mupai.studio           # 프로젝트마다 다를 수 있음
export UNSKEIN_PLANNER_TOKEN=<복사한 kind=planner 토큰>
export UNSKEIN_BUSINESS=<비즈니스 이름>                   # 서버 표기와 정확히 일치 — §5 repo_url 조회용 (예: ACME)
export UNSKEIN_PROJECT=<프로젝트 이름>                    # 서버 표기와 정확히 일치 (예: WebApp)
export UNSKEIN_SSH_KEY=${UNSKEIN_HOME}/creds/id_ed25519   # git 키(§4). HTTPS repo 면 이 줄 비움
```
- **6개 값**: `UNSKEIN_HOME`(맨 위) + `UNSKEIN_API` + `UNSKEIN_PLANNER_TOKEN` + `UNSKEIN_BUSINESS` + `UNSKEIN_PROJECT` + `UNSKEIN_SSH_KEY`. `UNSKEIN_BUSINESS`/`UNSKEIN_PROJECT` 는 **이름**으로, scope 가 이 이름을 서버에 조회해 `repo_url` 을 얻고 코드베이스를 clone 한다(§5) — 코드베이스가 없어도 프로젝트를 짚게 하는 게 이 두 변수의 취지(ADR-0022).
- ⚠️ **`export` 필수** — 없으면 `source planner.env && claude` 해도 값이 자식 claude 에 안 넘어간다(구 샘플의 잠복 버그, v1.20.0 에서 수정). (스킬의 `bin/planner-env.sh` 는 `set -a` 로 읽어 export 없이도 복구하지만, 명시 source 로 쓰려면 export 가 있어야 안전하다.)
- ⚠️ **`UNSKEIN_HOME` 은 옵션이 아니라 필수** — 이 한 줄이 상태 루트를 절대경로로 못박아, 어느 cwd 에서 source 해도 이 홈으로 확정되고 clone 위치(부모 아래)도 여기서 파생된다.

### 3.3 source 로 활성화 (source 우선) + cwd 폴백

**권장(명시 source)** — 프로젝트 작업폴더에서 claude 를 띄우기 전에 그 planner.env 를 source 한다. **코드베이스가 아직 없어도 된다**(scope 가 첫 clone 을 한다):
```shell
cd ~/work/webapp                              # 프로젝트 작업폴더(.unskein 의 부모)
source ~/work/webapp/.unskein/planner.env     # 그 프로젝트 전용 토큰·서버·프로젝트 이름
claude
#  → "이 범위로 스콥 떠줘" (unskein-scope) → 스콥이 코드베이스를 clone 하고 스콥한다(§5)
```
**폴백(자동) — 깜빡해도**: cwd 가 그 `.unskein` **아래(또는 그 부모)** 면, source 를 안 해도 스킬이 서버 호출 시 cwd 에서 위로 올라가 `.unskein/planner.env` 를 자동으로 찾는다. 코드베이스가 이미 clone 된 뒤에는 그 코드베이스 안에서 claude 를 띄워도(cwd 가 `.unskein` 의 형제라 부모에서 발견) 폴백이 작동한다.

- 확인: claude 세션에서 `unskein-scope`/`unskein-wbs` 를 처음 쓸 때 서버 도달로 검증된다. 어느 파일에서 토큰이 왔는지는 `unskein-doctor` 가 `UNSKEIN_PLANNER_ENV_FILE` 로 보여준다.
- 서버 스모크(선택): `source … && curl -s "$UNSKEIN_API/api/health"` → `{"status":"ok"}`.
- `401` = 토큰 오타 / kind 가 mori / 다른 서버 토큰 섞임.
- planner.env 를 고쳤으면 claude 를 껐다 다시(세션은 시작 시점 env 고정).
- **프로젝트마다 새 셸(또는 새 터미널)** — 이전 프로젝트의 `UNSKEIN_PLANNER_TOKEN` 이 셸에 남으면 source 우선 원칙상 그게 이겨 엉뚱한 프로젝트로 붙는다(§막히면). 헷갈리면 `env -u UNSKEIN_PLANNER_TOKEN -u UNSKEIN_API` 로 비우고 시작.

### 3.4 전역 `~/.unskein` 무력화 (프로젝트별 격리 필수)

cwd 폴백이 **대부분 경로의 조상인 전역 `~/.unskein`** 을 주울 수 있어, 프로젝트별 격리에서 전역이 남아 있으면 그림자로 가로챈다. 그래서 **전역을 무력화**한다:
```shell
mv ~/.unskein ~/.unskein_back      # 개명(되돌리기 가능). 확실히 지우려면 rm -rf
```
- 이후 프로젝트별 `.unskein/planner.env` 만 유효 — cwd 폴백이 조상 전역을 못 줍는다.
- bashrc 에 전역 planner.env 자동 로드 줄이 있으면 **제거**(프로젝트마다 명시 source 가 원칙).
- 전역 `~/.unskein` 폴백은 **하위호환용으로만** 남는다(전역 한 벌 방식은 ADR-0022 로 폐기). 멀티프로젝트에선 무력화가 맞다.

## 4. git 키 (`UNSKEIN_SSH_KEY`)

scope 가 코드베이스를 clone/pull 할 때 쓰는 git 인증이다. **HTTPS 냐 SSH 냐는 서버의 `repo_url` 형식이 정한다**(내 취향이 아니라):

- **SSH repo**(`git@…`/`ssh://…`) → **`UNSKEIN_SSH_KEY`** 로 붙는다. scope 가 `GIT_SSH_COMMAND="ssh -i $UNSKEIN_SSH_KEY -o IdentitiesOnly=yes"` 로 이 키를 쓴다(실행기와 **같은 변수명** — EXECUTOR 도 SSH repo 를 `UNSKEIN_SSH_KEY` 로 붙는다). 키를 만들어 공개키를 GitHub 에 등록한다:
  ```shell
  ssh-keygen -t ed25519 -f "$PROJ/.unskein/creds/id_ed25519" -N ""   # → .pub 을 GitHub 에 등록
  #   (계정 키 또는 그 repo 의 deploy key). 표준 위치라 planner.env 의 UNSKEIN_SSH_KEY 기본값과 일치.
  ```
  키는 상태 루트 밖·권한 600·코드베이스 밖(커밋에 안 섞임 — 실행기와 같은 이유). PLANNER 의 `.unskein/` = `planner.env` + `creds/`(실행기의 `work/` 는 없음 — 다오가 없으니).
- **HTTPS repo**(`https://…`) → SSH 키 대신 **`gh` 자격**(`gh auth setup-git`, §1-A ③)에 맡기고 **`planner.env` 의 `UNSKEIN_SSH_KEY` 줄은 비운다**. 한 계정으로 모든 repo 가 되면 이게 가장 단순하다.

문서 PR 용 GitHub 인증(`gh pr create`)은 `gh auth` 가 담당한다 — planner.env 에 넣지 않는다.

> **`Permission denied (publickey)`** — SSH repo 인데 clone 이 이걸로 실패하면, 그 `UNSKEIN_SSH_KEY` 의 공개키가 GitHub(계정 또는 deploy key)에 등록됐는지 확인한다. 등록이 어려우면 HTTPS repo 로 두고(서버 `repo_url` 을 `https://…` 로) `UNSKEIN_SSH_KEY` 를 비워 gh 자격으로 붙는다. scope 는 clone/인증 실패를 **강제하지 않고 멈추고 보고**한다(§5, fallback 금지).
> (선택) 커밋 신원을 프로젝트별로 두려면 clone 된 코드베이스에 `git config user.name/user.email` 을 건다.

## 5. 자동 clone/pull 로 코드베이스 보장 (ADR-0022)

**사람이 수동으로 `git clone`/`git pull` 하지 않는다.** 스콥의 근거는 코드베이스인데, 그 코드베이스를 최신으로 보장하는 일을 `unskein-scope`(§0 "코드베이스 보장")가 **스콥 시작 전에 스스로** 한다 — 없으면 clone, 있으면 `git pull --ff-only`. 최초 clone 도 여기서 한다.

**scope §0 이 도는 순서**:

1. **인증 로드** — 서버·git 호출 전에 `bin/planner-env.sh` 로 셸에 올린다(source 우선·cwd 폴백). `UNSKEIN_PLANNER_TOKEN` 이 없으면 **멈추고 알린다**(fallback 금지 — 다른 토큰으로 안 넘어감).
2. **repo_url 조회** — 플래너 토큰은 유저(멤버십)까지만 특정하지 어느 프로젝트인지 못 짚고, 코드베이스가 없으면 `git remote` 로도 알 수 없다. 그래서 `planner.env` 의 **`UNSKEIN_BUSINESS`/`UNSKEIN_PROJECT` 이름**으로 서버에서 `repo_url` 을 조회한다(이름 단건 라우트가 없어 목록을 받아 `name` 정확 일치로 고른다):
   ```
   GET /api/businesses                     (X-Planner-Token)  → UNSKEIN_BUSINESS 이름 일치 → biz_id
   GET /api/businesses/{biz_id}/projects   (X-Planner-Token)  → UNSKEIN_PROJECT 이름 일치 → {id, repo_url}
   ```
   둘 다 `X-Planner-Token` 으로 닿는 인가 경로다(ADR-0013). 여기서 얻은 `id` 가 §등록 대상 `project_id` 다.
3. **코드베이스 보장** — clone 위치는 상태 루트의 **부모** 아래 `repo basename`:
   ```
   CODEBASE = $(dirname "$UNSKEIN_HOME")/<repo basename>
   #   예: UNSKEIN_HOME=/home/u/work/webapp/.unskein · repo_url=…/WebApp.git
   #       → CODEBASE=/home/u/work/webapp/WebApp
   # SSH 키가 있으면 GIT_SSH_COMMAND="ssh -i $UNSKEIN_SSH_KEY -o IdentitiesOnly=yes"
   #   있으면:  git -C "$CODEBASE" pull --ff-only
   #   없으면:  git clone "$REPO_URL" "$CODEBASE"
   ```
   이후 `cd "$CODEBASE"` 하고 그 코드베이스를 근거로 스콥한다.

**멈추고 보고하는 지점 (fallback 금지, ADR-0022)**:
- `git pull --ff-only` 실패(로컬 커밋이 앞섬·충돌·detached) → **강제하지 않는다**(`reset`·`-f` 금지 — 미푸시·다른 세션 작업 유실 위험). 상태를 그대로 보고하고 멈춘다.
- clone/인증 실패(권한·키), 비즈니스/프로젝트 이름 미매칭(오타·미가입·미등록), `repo_url` 공란(프로젝트에 원격 미설정) → 임의로 넘어가지 않고 멈추고 사람에게 묻는다. `repo_url` 미설정이면 확인 후 채워 다음부터 자동 되게 한다.
- **정합 확인**: 이미 있던 코드베이스면 `git remote get-url origin` 이 조회한 `repo_url` 과 정규화 후 일치하는지 본다(`.git` 접미 제거·`https`↔`ssh` 동치). 다르면 엉뚱한 repo 이니 멈추고 묻는다.

> **역매핑은 하위호환 폴백으로만 남는다**: 종전엔 코드베이스가 이미 있을 때 `git remote get-url origin` 을 서버 프로젝트 `repo_url` 목록과 대조해 프로젝트를 역매핑했다. 이제 기본은 `UNSKEIN_BUSINESS`/`UNSKEIN_PROJECT` 이름으로 프로젝트를 **직접 짚는 것**이고, 역매핑은 이름 변수 없이 코드베이스만 이미 있을 때의 하위호환 경로다(정확히 하나 일치일 때만). 새 설치는 이름 변수를 쓴다.

## 6. 운영자 워크플로 — 예시로 보는 실제 흐름

플래너 사용자에게 기대하는 건 **개발이 아니라** "작업을 검증 가능한 기준으로 쪼개 서버 큐에 등록"하는 것이다. 하나의 의뢰가 어떻게 흘러가는지 **구체 예시**로 본다. (플래너 세션 = `~/work/webapp` 에서 source 후 띄운 `claude`. 굵은 화살표 `»` 는 사용자가 세션에 입력하는 말.)

> **달리는 예시 의뢰**: *"작업 목록 화면에 상태별(대기/진행/완료) 필터를 추가해줘."* (대상 프로젝트 = WebApp)

### ① 의뢰 받아 이해하기 — "무엇을, 어디를 건드리나"

먼저 코드베이스 근거로 범위를 파악한다. (첫 스콥이면 이 시점에 scope §0 가 코드베이스를 clone 한다 — §5.) 기존 지식·구조부터 확인해 **재분석을 피한다**.
```
» 작업 목록 화면에 상태별 필터를 추가하려고 해. 먼저 관련 코드랑 이미 있는 지식부터 찾아줘.
```
- 플래너가 하는 일: `unskein-wiki-search`(repo 기존 결정·주의사항) + understand 그래프/코드 탐색으로 **어느 파일이 걸리는지**(작업 목록 API·프론트 컴포넌트·상태 enum) 파악.
- 결과 예: "상태 필드는 backend `tasks` 모델의 `status`(대기/진행/완료), 목록은 `GET /api/tasks`, 화면은 `TaskList.tsx`. 필터 쿼리 파라미터는 아직 없음."

### ② 스코프 — `unskein-scope` (핵심 작업)

파악한 걸 **검증 가능한 수용 기준**으로 내리고, 사람과 점검하고, WBS로 쪼갠다.
```
» 이 범위로 스콥 떠줘.  (→ unskein-scope 트리거)
```
플래너가 만드는 산출물 예시:
- **수용 기준**(`docs/specs/`, "됐다"를 기계적으로 판정 가능하게):
  - `GET /api/tasks?status=진행` 이 해당 상태만 반환한다.
  - 화면 상단 필터 칩(대기/진행/완료/전체) 클릭 시 목록이 즉시 걸러진다.
  - 잘못된 status 값은 400 을 반환한다.
- **사람과 점검**(drive-then-ask): "필터를 URL 에 남길까요(공유가능), 아니면 화면 상태로만?" 처럼 **갈리는 지점만** 묻는다.
- **WBS 분해**(`logical-thinking` 위임, 행위주체 기준):
  ```
  1. backend: GET /api/tasks 에 status 쿼리 필터 추가
  2. backend: 잘못된 status 값 400 처리
  3. frontend: TaskList 상단 필터 칩 UI
  4. frontend: 칩 클릭 → 목록 재조회 연결
  ```

### ③ 등록 — 각 카드에 `plan_doc` 갖춰 서버 큐로 (`unskein-scope` + `unskein-wbs`)

WBS 각 항목을 **서버 프로젝트 작업 큐에 카드로 등록**한다. 카드마다 수용 기준(`plan_doc`)이 붙어야 한다.
```
» 이 WBS 를 WebApp 프로젝트에 등록해줘.  (→ unskein-wbs)
```
- **프로젝트 확정**: 등록 대상 `project_id` 는 §5 ②에서 `UNSKEIN_BUSINESS`/`UNSKEIN_PROJECT` 이름으로 이미 특정됐다. (하위호환: 이름 변수 없이 코드베이스만 있으면 `git remote` 역매핑 — 정확히 하나 일치일 때만, 모호하면 사람에게 물음. 임의 선택 금지.)
- **등록 메커니즘**: 생성 API(`POST /projects/{id}/tasks`)는 `plan_doc` 을 **안 받는다** → 등록은 **(a) 카드 생성 → (b) `plan_doc` 부여** 두 동작이다:
  - backlog 로 두고 기록만: `PATCH /tasks/{id}` 에 `plan_doc`.
  - 실행대기로 올리며 부여: `POST /tasks/{id}/plan`(`attach_plan`) — `plan_doc` 저장 + `backlog→plan`. **빈 `plan_doc` 은 게이트가 거부**(ADR-0009) — "등록만 하고 기준은 나중"이 안 된다.
- 결과 예: 4개 카드가 `plan` 상태로 큐에 올라감.

### ④ 큐에 넘김 — 여기서 플래너 손을 뗀다

- 등록된 카드는 큐에서 **EXECUTOR 가 claim** → **다오(WSL)** 가 클론·구현·PR 한다. **플래너는 개발에 관여하지 않는다.**
- 다오가 막혀 **질문(`QUESTION`)** 을 던지면 웹에서 사람이 답하고, 다오가 같은 세션을 이어받아 계속한다(대화턴). 이 답변도 플래너가 아니라 **의뢰/운영 담당**이 웹에서 한다.

### ⑤ 마감·운영 — 문서 변경 PR·인계·진단

플래너 자신이 만든 문서·스펙 변경이 있으면 마감한다.
```
» 스콥 문서 변경분 커밋하고 PR 올려줘.  (→ unskein-git)
» 이 세션 내용 다음 세션에 넘기게 정리해줘.  (→ unskein-session)
```
- `unskein-git`: feature 브랜치 + 커밋 + PR. **기본 브랜치 직머지 금지**(사람 리뷰 게이트).
- `unskein-session`: 세션 요약을 `docs/local/` 에 남겨 다음 세션이 복원.
- 그 외: 배포 점검 `unskein-deploy`, 작업 점유 문제 `unskein-lock`, 환경 진단 `unskein-doctor`, 화면 검증 `unskein-test`.

> **한 눈에**: 의뢰(①) → **수용 기준·WBS(②)** → **plan_doc 갖춰 등록(③)** → 큐로 넘기고 손 뗌(④) → 문서 마감·인계(⑤). ②③ 이 플래너의 본업이고, 실제 코딩·배포는 실행 측(EXECUTOR→다오)이 가져간다.

## understand-anything (코드 이해 그래프, ADR-0012)

understand-anything(Egonex-AI, MIT, v2.8.1)은 코드베이스를 **지식 그래프**로 만들어 파악·온보딩·변경영향 분석에 쓰는 도구다. 대상 프로젝트 코드베이스에 그대로 적용한다.

> ### ✅ 핵심: 팀에서 **딱 한 명만** 하면 된다
> 그래프는 **단일 거대 JSON** 파일이라, 여러 명이 각자 만들면 커밋이 서로 충돌한다. 그래서 **한 명이 그래프를 만들어 repo 에 커밋(=배포)** 하고, 나머지 팀원과 다오는 **아무것도 안 하고 `git pull` 로 그 그래프를 받아 쓴다**.
>
> → **당신이 그 담당자가 아니면 이 절 전체를 건너뛰어도 된다.** 이미 커밋된 그래프가 `git pull`(=§5 자동 pull) 로 자동으로 따라온다. (다오는 애초에 대상이 아니다 — 구조적으로 제외됨.)

### 담당자 한 명이 하는 일 (5단계)

**1. 런타임 준비** — node≥22 · pnpm≥10 · python3 · git 이 있으면 된다. (§1-A 를 따랐으면 이미 갖춰짐. pnpm 만 없으면 `corepack enable pnpm` 한 번.)

**2. 플러그인 설치** — 코드베이스에서 `claude` 를 **새로 띄우면** "이 저장소가 요구하는 플러그인을 설치할까요?" 안내가 뜬다 → **동의**하면 끝.
```shell
claude plugin list        # understand-anything@understand-anything 보이면 설치 완료
```
> 안내가 안 뜨면 수동으로: `claude plugin marketplace add Egonex-AI/Understand-Anything` → `claude plugin install understand-anything`.

**3. 분석 실행** — 핵심 폴더 하나를 스콥해 그래프를 만든다:
```shell
/understand-anything:understand <핵심폴더> --no-auto-update
```
지킬 것 3가지만 기억하면 된다:
- **`--no-auto-update` 를 꼭 붙인다** (자동 갱신을 켜면 그래프를 제멋대로 커밋해 다른 세션과 충돌한다).
- **git worktree 안에서 돌리지 않는다** (엉뚱한 위치에 그래프가 써진다). 그냥 코드베이스 본체에서.
- **한 번에 폴더 하나** (파일이 많으면 도구가 알아서 나눠 하자고 안내한다).

**4. 품질 확인** — 분석이 끝나면 대시보드가 자동으로 뜬다. 구조·연관검색이 쓸 만한지 보고, `/understand-chat "<질문>"` 이 관련 주변 코드까지 끌어와 답하는지 확인한다. (`.understand-anything/config.json` 에 `"autoUpdate": false` 인지도 한 번 확인 — true 면 끄고 다시.)

**5. 커밋 = 배포** — 쓸 만하면 `.understand-anything/` 폴더를 **커밋·push** 한다. 이게 곧 팀 배포다 — 이때부터 모두가 `git pull` 로 그래프를 받는다. (임시·휘발 파일은 `.gitignore` 가 알아서 뺀다.)

> 🔒 **private 주의**: 분석은 소스를 host LLM(Claude 기본 모델)에 보낸다 — 평소 Claude Code 쓸 때와 동일한 노출이다. 단 **3rd-party 모델(README 의 Ollama 등)로는 돌리지 말 것.**

## 동시 세션 규율 (필수)

PLANNER 는 여러 세션이 **같은 코드베이스(working tree)를 동시에** 쓸 수 있다. working tree 는 하나·브랜치도 하나라, 공유 트리에서 `git checkout`·`reset`·`commit`·sync 를 하면 **다른 세션을 작업 중 브랜치에서 빼버린다**(1순위 사고). scope 의 `git pull --ff-only`(§5)도 코드베이스 쓰기라 여기 해당한다 — 다른 세션이 그 트리에서 커밋 중이면 pull 이 멈추고 보고한다(강제 안 함).

- **세션 시작 점검**: `ps -eo pid,etime,args | grep [c]laude` + `git worktree list` + `find <repo> -mmin -10 -not -path '*/.git/*'` 로 누가 어느 브랜치·파일을 점유 중인지 확인.
- **쓰기가 필요하면 `git worktree add -b <브랜치> <경로> origin/<기본브랜치>` 격리본**에서 편집·커밋·PR. 공유 트리의 브랜치·파일은 안 건드린다.
- **읽기·API 조회·스코프 등록은 트리 무관이라 예외** — PLANNER 의 핵심 작업(등록)은 대부분 여기 해당해 공유 트리에서 그대로 해도 된다. 정작 주의할 건 코드·문서를 **커밋**할 때다.
- `docs/local/` 은 (해당 프로젝트가 gitignore 해 두면) 로컬 전용이라 새 파일 기록은 git 영향 없음.

## 안전 규칙 (프로젝트별 격리)

| 규칙 | 위반 시 |
|---|---|
| **① 프로젝트마다 전용 planner 토큰**(kind=planner) — 공유 금지 | 유출 시 피해가 전 프로젝트로 번짐. mori 토큰과 섞으면 401 |
| **② `.unskein`(planner.env + creds 키) 은 코드베이스 repo 밖** — `<작업폴더>/.unskein/`(코드베이스는 그 옆) | repo 안에 두면 토큰·git 키가 커밋·`git clean` 에 섞일 위험 |
| **③ 프로젝트마다 새 셸 + 전역 무력화 + 폴더 비중첩** | 셸에 남은 이전 토큰이 source 우선으로 이겨 오연결 / cwd 폴백이 조상 `.unskein`(또는 전역)을 주움 → **엉뚱한 프로젝트로 조용히 붙음** |
| **④ 비즈니스/프로젝트 이름 정확 일치**(서버 표기와) | 미매칭이면 scope 가 `repo_url` 을 못 얻어 멈추고 묻는다(fallback 금지) — 오타·미가입·미등록 |

권장: **프로젝트별 세션 + 전용 토큰 + 프로젝트 옆 `.unskein/planner.env`(코드베이스 밖) + 전역 무력화**. 배경 설계 결정: ADR-0021·0022.

## 막히면

> 스킬은 셸 명령이 아니라 **claude 세션 대화에서 이름으로 호출**("unskein-doctor 로 진단해줘").

- 환경·연결 진단 → `unskein-doctor`. `UNSKEIN_PLANNER_ENV_FILE` 로 **어느 planner.env 에서 토큰이 왔는지** 확인(엉뚱한 프로젝트로 붙었나) + 프로젝트 격리 모드에서 전역 `~/.unskein` 잔존 경고.
- **401** → 토큰 kind(mori 를 planner 자리에) / 다른 서버 토큰 섞임 / 오타.
- **엉뚱한 프로젝트로 붙음** → 셸에 이전 토큰 잔존(source 우선이 이김) → **새 터미널**에서. + cwd 폴백이 조상 전역을 주웠나 → 전역 `~/.unskein` 무력화(§3.4).
- **scope 가 프로젝트를 못 짚음(이름 미매칭·`repo_url` 공란)** → `UNSKEIN_BUSINESS`/`UNSKEIN_PROJECT` 가 서버 표기와 정확히 같은지, `repo_url` 이 채워졌는지 확인. 임의 선택 금지.
- **clone 실패 `Permission denied (publickey)`** → SSH repo 인데 `UNSKEIN_SSH_KEY` 공개키가 GitHub 에 미등록 → 등록하거나 HTTPS repo + gh 자격으로(§4).
- **`git pull --ff-only` 실패** → 로컬 커밋·충돌 — scope 가 강제 안 하고 멈춘다. 그 코드베이스에서 미푸시 작업/충돌을 사람이 정리(§동시 세션 규율).
- `source planner.env` 했는데 스킬이 토큰을 못 봄 → planner.env 값에 **`export`** 가 있는지(구 샘플 버그 — §3.2).
- 화면 런타임 검증 → `unskein-test`(CDP, 윈도우 실환경은 별도).
