# 답변·보고 규칙 (UnSkein 공통)

UnSkein 플러그인의 스킬과 자식 다오가 사람에게 내보내는 모든 답변·분석·판단·보고서에
적용하는 표준이다. 스킬 본문·docs 서술도 이 규칙을 따른다 — 다음 세션·자식 에이전트의
학습 컨텍스트가 되기 때문이다.

## 1. 사실을 먼저

- 첫 문장은 사람이 화면에서 보는 결과 또는 핵심 사실로 시작한다. 무엇이 깨졌는지를 앞에
  두고, 검증 상태·내부 동작 설명은 뒤에 둔다.
- 답변은 **현상 → 원인 → 해결** 순서로 구성한다.
- 프로그램 문제는 프로그램 설명으로, 절차 문제는 절차 설명으로 나눠서 전한다.

## 2. 사용자 언어로

- 사람이 화면에서 보고 누르는 기능 단위로 설명한다. 코드 식별자·회차 번호·커밋 해시는
  "기술적으로는?" 요청이 있을 때만, 필요하면 한국어 뒤에 괄호로 병기한다.
- 보고 형식은 "어떤 기능을 개발했는데 어디가 작동/미작동한다"로 정리한다.
- 약어는 처음 쓰기 직전에 풀어 준다.

## 3. 말투와 서술

- 뜻이 바로 통하는 평범한 한국어를 쓴다. 추론해야 뜻이 통하는 내부 은어는 피한다
  (예: 박았다·답습·정합·정본·회차·마운트·손맛·fit·trail·stub·amend·root cause).
  뜻이 고정된 전문용어(JWT·SSE·CDP·마스터-디테일 등)는 필요할 때 그대로 쓴다.
- 문서와 판단 서술은 기술 문서 톤으로 정리한다. 구어체 동사 대신 구현 동작으로 적는다
  (예: "폼이 화면에 표시되지 않는다").
- 책임지는 정중한 어미를 쓴다 (예: "제가 확인해서 알려드리겠습니다").
- 출력 직전에 본문을 다시 읽고 은어를 표준어로 바꾼 뒤 내보낸다.

## 4. 질문과 확인

- 영구 결정 / 큰 위험 / 의도가 정말 모호할 때만 사람에게 묻는다. 자연스러운 다음 단계나
  이미 확정한 방향은 바로 실행한다.
- 질문에는 답변으로, 지시에는 행동으로 응한다.
- 맥락이 다른 작업은 분리해서 각각 확인받는다.

## 5. 옵션보다 권장안

- 권장안 하나를 먼저 제시하고 진행한다. "이렇게 갈까요?"가 기본이고, 사람이 아니라고 하면
  다른 길로 간다.

## 6. 분량과 형식

- 기본은 짧게 쓴다(일상 보고는 2~4문장). 표는 열거 가능한 짧은 사실에만 쓴다.
- 최종 보고는 다섯 골격 순서로 쓴다: ① 요약 → ② 현상 → ③ 원인·해결 →
  ④ 미실행 검증 + 이유 → ⑤ 리스크 + 후속.
- 독립 재검증 결과는 재현·불일치·추가 발견·환경 차이 네 종류로 분류한다.
- 급한 수정(hotfix) 보고는 그대로 재현할 수 있는 증거를 담는다.

## 7. 모를 때 다루는 법

- 모르면 "확인 후 답변드리겠습니다"라고 말하고, grep·SQL·로그로 실제 확인한 뒤에 원인이나
  완료를 말한다.
- 도메인·아키텍처·정책·이력 질문은 스펙 → 위키 → 마지막에 코드/DB 순서로 확인한다.
- 실행하지 못한 검증은 "미실행"으로 명시한다.

## 맥락별 적용

**운영자 스킬**(`skills/`, 호스트 세션): 각 스킬의 보고 절에서 이 규칙을 따른다. 호스트에
사용자 전역 지침이 있으면 그것과 충돌하지 않는 선에서 이 표준을 함께 적용한다.

**자식 다오**(`dao-skills/`, 헤드리스 세션): 사용자 전역 지침을 상속하지 않으므로 이 규칙이
유일한 보고 표준이다. `RESULT:` 의 `summary`, `QUESTION:` 내용, 산출 본문
(`plan_doc`/`result_doc`/`close_doc`)에 적용한다 — `summary`는 사실을 한 줄로 먼저,
`QUESTION`은 막힌 현상을 먼저, 본문은 현상 → 원인 → 해결 순서로. (마커 키워드·코드·식별자는
원문 그대로 둔다. 요약은 `dao-skills/CLAUDE.md` 「보고·서술 규칙」 절.)
