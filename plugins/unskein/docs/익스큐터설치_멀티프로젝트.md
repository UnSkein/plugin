# 실행기(모리 EXECUTOR) 설치 — 멀티프로젝트 (WSL-first)

> 이 문서는 UnSkein 플러그인과 함께 배포되는 일반 설치 가이드다. (v1.27.0)

> 윈도우 호스트 위에 **모리 EXECUTOR** 를 세우는 절차 — **한 머신에서 여러 프로젝트**를 처리(멀티프로젝트)까지. EXECUTOR = UNSKEIN(PM SaaS) 큐를 폴링해 작업을 선점(claim)하고 **다오(`claude -p`)를 구동**(`run_once`/`run_loop`). **스코프·계획은 안 한다**(PLANNER 몫) — `plan_doc` 이 붙은 카드를 받아 구현한다.
>
> 🔑 **WSL-first (핵심 전제)**: 실행기의 **개발 처리는 전부 윈도우 안의 WSL distro 내부에서 완결** — 플러그인·env·creds·work·빌드런타임이 모두 그 distro 안. 윈도우의 역할은 ①distro 호스팅 ②운영자가 실행기를 **수동** 기동 ③브라우저 화면검증(CDP)뿐.
>
> 🔑 **멀티프로젝트 = 두 방식**(§1에서 선택): **A. 한 watch·비즈니스 범위**(설계 결정 ADR-0006, 내부 동시성) / **B. 프로젝트별 watch + `UNSKEIN_HOME` 상태 격리**(설계 결정 ADR-0020, **plugin ≥ 1.18.0**).
>
> 🔑 **상태 로드 = source 우선·cwd 폴백**(**plugin ≥ 1.20.0**, 설계 결정 ADR-0021 — 플래너와 통일): `source <프로젝트>/.unskein/executor.env` 가 늘 이기되(source 우선), 깜빡해도 그 `.unskein` 아래에서 띄우면 `run_once` 가 cwd 에서 위로 `executor.env` 를 자동 로드한다. 실측에서 source 누락으로 연결이 안 되는 사례가 있어 cwd 폴백으로 견고화했다. 단 **셸에 이전 프로젝트 토큰이 남으면 그게 이기니(source 우선) 프로젝트마다 새 셸** + **전역 `~/.unskein` 무력화**(§5·§7).
>
> ⚠️ PLANNER 설치는 짝 문서 → 같은 폴더의 `플래너설치.md`. 윈도우 네이티브(다오 직접 구동)는 미실증 향후 트랙 → 부록 A.

## 0. EXECUTOR vs PLANNER — 무엇을 세우는가

| | **EXECUTOR** (이 문서) | **PLANNER** (짝 문서) |
|---|---|---|
| 정체 | WSL distro 안 `run_once`/`run_loop` 폴링 실행기 | 개발자의 Claude Code / 운영 세션 |
| 코드베이스 | **없음** — 다오가 작업마다 클론 | **보유**(git pull 최신화) |
| 하는 일 | 큐 claim → **WSL 안 다오** 구동 → 개발 → PR | 의뢰 → 스코프·플랜 → 타스크 등록 |
| 쓰는 것 | `/unskein:run`·`/unskein:watch`·`/unskein:status` + 다오 스킬 6개(작업마다 자동 이식) | 운영자 스킬 `unskein-scope`·`unskein-wbs` 등 |
| 서버 접근 | **모리 토큰**(kind=mori, `X-Mori-Token`) | planner 토큰 또는 admin 로그인 |

핵심: **EXECUTOR 설치 = "WSL distro 안에 플러그인 + kind=mori 토큰 + git 클론 자격증명 + watch 범위 + 빌드런타임"을 갖춘 상주 프로세스를 세우는 것.** 셋업 전 과정은 **`unskein-setup` 스킬 하나**가 안내한다(S0 배치 모드 → S1 연결·인증 → S2 프로비저닝 → S3 검증). 막히면 `unskein-doctor`.

## 1. 멀티프로젝트 두 방식 — 먼저 고른다

| | **A. 한 watch — 비즈니스 범위** | **B. 프로젝트별 watch — `UNSKEIN_HOME` 격리** |
|---|---|---|
| 근거 | 설계 결정 ADR-0006 (조율자 내부 동시성) | 설계 결정 ADR-0020 (상태 루트 격리) · **plugin ≥ 1.18.0** |
| watch 세션 | 1개 (`bis` 만 지정, `prj` 비움) | **프로젝트마다 1개** (각자 `bis`+`prj`) |
| 상태(env·creds·work) | 전역 `~/.unskein` 한 벌 | **프로젝트마다** `<프로젝트>/.unskein` 한 벌 |
| 모리 토큰 | 1개 | **프로젝트마다 전용 발급** (공유 금지 — §8 ②) |
| git 자격증명 | 하나가 전 repo 커버해야(§5 커버리지) | 프로젝트별 분리 가능(org 흩어져도 OK) |
| 정책(`AUTO_ADVANCE_TEST`·`AUTO_MERGE`·모델/effort) | 전 프로젝트 일괄 | **프로젝트별** |
| 프로젝트 간 병렬 | 한 루프 안 스레드 풀(`UNSKEIN_MAX_CONCURRENCY`) | 세션 수 만큼 (각 세션 안에서 또 동시성) |
| 장점 | 단순 — 터미널 1개, env 1벌 | 격리 — 한 프로젝트의 토큰·정책·자격증명·사고가 다른 프로젝트로 안 샘 |
| 고르는 기준 | 같은 비즈니스·같은 정책·같은 org repo 면 충분 | 정책/자격증명/서버가 갈리거나, 프로젝트별로 켜고 끄고 싶을 때 |

- **B 의 코드·상태 축**: 코드(플러그인)는 **전역 1벌 공유**(업데이트 1곳), 상태만 `UNSKEIN_HOME` 밑으로 갈라진다. `UNSKEIN_HOME` 미설정 = 전역 `~/.unskein`(A 와 동일, 하위호환).
- 혼합도 된다: 비즈니스 X 는 A 로 한 세션, 별도 정책이 필요한 프로젝트 Y 만 B 로 격리.

### 동작 원리 (공통)

```
 [WSL distro 안]                          A: 세션 1개          B: 세션 N개(프로젝트별)
   모리 EXECUTOR (run_loop = 단일 조율자, 세션마다 1개)
   큐 폴링 → claim(watch 범위) → 다오 풀에 분배 (UNSKEIN_MAX_CONCURRENCY, 기본 3)
      ▼                     ▼
  다오 (WebApp)        다오 (WebAPI)       ← 전부 이 distro 프로세스
  WORK_ROOT/101/…      WORK_ROOT/102/…    ← 작업별 폴더 격리 (B 는 WORK_ROOT 자체가 프로젝트별)
  개발 → git PR        개발 → git PR
```

- **이중 선점 방지 = 서버**(`FOR UPDATE SKIP LOCKED` + heartbeat 신선도 180s, 작업별 heartbeat 60s). 클라이언트 single-flight 는 프로세스-내 전용.
- **프로젝트 안 순차, 프로젝트 간 병렬**: claim 게이트(의존 간선 + WBS 서브트리 잠금)는 프로젝트 경계를 넘지 않는다. 단 "안 순차"는 **의존 간선이 등록된 만큼만**(간선 없는 독립 카드는 동시 선점 — PLANNER 가 간선 빠짐없이).
- **알려진 한계**: ① 모델·effort 는 셸 env 상속이라 **그 세션의 모든 다오 공유** — 루프 전 `env -u CLAUDE_EFFORT -u ANTHROPIC_MODEL` 정리 권장(B 는 세션별이라 프로젝트별로 다르게 줄 수 있음). ② 한 프로젝트에 게이트 탈락 후보 100+ 이면 같은 세션의 다른 프로젝트 선점 지연 가능(A 의 한계 — B 는 무관). ③ same-repo 작업 2건 동시면 PR 충돌 가능 — 의존 간선이 실질 방어선.

## 2. 공통 사전 준비 — WSL distro 안

> **모든 명령은 WSL distro 셸 기준.** `wsl -d <distro>` 로 들어가 진행. 아무것도 없는 신규 distro 는 아래 2-A 부터.

| 도구 | 왜 | 확인 |
|------|----|----|
| `claude` (Claude Code) | 다오 + 플러그인 호스트 | `claude --version` |
| `git` | 다오의 클론·push | `git --version` |
| `gh` (**인증 필수**) | 마감(`unskein-git`) PR 생성 — **치명 preflight**(미인증이면 claim 시작 안 함) | `gh auth status` |
| `python3` | 오케스트레이터 | `python3 --version` |
| **빌드 런타임** | 다오가 WSL 안에서 클론→빌드→테스트 | 프로젝트 스택별(node·pnpm 등) |

- gh: `gh auth login`(대상 repo push/PR 권한 계정) → `gh auth setup-git`.
- 빌드 런타임이 없으면 다오가 검증 단계에서 정직하게 `QUESTION` 회수(버그 아님).
- **watch 는 distro 상주 전제** — distro 가 내려가면 루프도 죽는다. 재기동 절차는 §6.

### 2-A. 신규 distro 부트스트랩

```shell
# ① Claude Code 설치 + PATH + 로그인
curl -fsSL https://claude.ai/install.sh | bash
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc && source ~/.bashrc
claude --version && claude        # 첫 실행 → Anthropic 계정 로그인
```
**② 베이스 런타임 — 아래 프롬프트를 claude 에 붙여넣어 시킨다**(인증만 사람 몫):
```text
이 WSL 배포판을 unskein 실행기로 클린 셋업한다. 먼저 `uname -a` 와 배포판을 확인하고,
이 OS 패키지 매니저로 아래를 설치·검증해줘. 이미 있으면 건드리지 말고 버전만 출력해
(fresh distro 라 대부분 신규 설치일 것):
- git
- python3
- gh (GitHub CLI) — 설치까지만. 인증(gh auth login)은 내가 직접 한다.
- 대상 프로젝트 스택의 빌드 런타임 — 다오가 WSL 안에서 클론→빌드→테스트 하는 데 필요.
  어떤 스택(node/pnpm·python venv·go 등)인지 물어보고 그에 맞춰 설치.
각 도구는 설치 후 버전을 출력해 확인하고, sudo·전역 설정 변경이 필요하거나 실패하면
임의로 진행하지 말고 멈춰서 물어봐. (wslu 가 있으면 로그인 URL 을 윈도우 브라우저로 여니 편하다.)
```
**③ 인증은 사람이 직접**: `gh auth login && gh auth setup-git` (`wslu` 설치 시 로그인 URL 이 윈도우 브라우저로).

## 3. 플러그인 설치 — WSL claude 에 (user 스코프, 전역 1벌)

```shell
claude plugin marketplace add UnSkein/plugin
claude plugin install unskein@unskein-marketplace
claude plugin list        # Scope: user / 버전 확인 — 방식 B 는 v1.18.0 이상 필수
```

- **버전이 낡게 잡히면**: `claude plugin marketplace update unskein-marketplace` → `claude plugin update unskein@unskein-marketplace` → **claude 재시작**.
- 방식 B 를 쓰는데 1.18.0 미만이면 `UNSKEIN_HOME` 이 통째로 무시된다(전역 `~/.unskein` 으로 동작) — 반드시 버전부터.
- 설치본 위치: `~/.claude/plugins/cache/unskein-marketplace/unskein/<버전>/` (orchestrator·dao-skills·templates·bin). 다오 스킬은 따로 설치할 게 없다 — 모리가 **작업마다** `WORK_ROOT/<task_id>/` 에 심는다.

## 4. 셋업 — `unskein-setup` 스킬 하나로 (S0→S3)

> WSL 안 `claude` 세션에서 **"unskein-setup 으로 셋업해줘"** — 아래 전 과정을 스킬이 안내한다. 수동으로 해도 결과는 같다(상세는 4.1~4.3). **방식 B 는 반드시 프로젝트 디렉토리에서** 스킬을 실행(그 `$PWD/.unskein` 을 상태 루트로 제안받는다).

| 단계 | 내용 | 방식 A | 방식 B (프로젝트마다 반복) |
|------|------|--------|---------------------------|
| **S0** | 배치 모드·상태 루트 확정 | `UNSKEIN_HOME` 미설정(전역 `~/.unskein`) | `UNSKEIN_HOME=<프로젝트>/.unskein` — **절대경로**로 executor.env 맨 위에 기록 |
| **S1** | 연결·인증 — `executor.env` 한 파일 + 폴더 생성(`mkdir -p creds work`) | env 1벌, 토큰 1개 | **프로젝트별 env + 전용 토큰**(§8 ②) |
| **S2** | 런타임 설치 + **실클론 검증** + 의존성 | 스코프 전 repo | 그 프로젝트 repo |
| **S3** | preflight 통과 확인("즉시 처리 가능" 게이트) | 공통 | 공통 — `[경로]` 줄로 상태 루트 확인 |

### 4.1 서버 연결 — 모리 토큰 (S1 상세)

실행기가 큐에 붙는 데 필요한 건 `UNSKEIN_API` + `UNSKEIN_MORI_TOKEN` 둘. 실행기(python)는 **자기 셸의 환경변수**만 읽는다(파일 자동 로드 없음) — 정본은 **executor.env 에 담아 `source` 한 셸에서 `claude` 를 띄우기**.

1. **계정·멤버십**(관리자에게): 웹 로그인 계정 + 담당할 비즈니스·프로젝트 멤버십.
2. **웹에서 모리 토큰 발급**: 서버(예: `https://unskein.mupai.studio`) 로그인 → 설정에서 발급 — **kind 반드시 `mori`**(⚠️ planner 아님). **딱 한 번 보임 — 즉시 복사.** 토큰 이름은 실행기(방식 B 는 프로젝트) 전용으로(예: `office-webapp`).
3. **executor.env 작성**(chmod 600) — 템플릿 복사 후 **편집기로** 채운다(echo/명령줄 금지 — 셸 기록에 남음):
```shell
TPL=$(find ~/.claude/plugins/cache/unskein-marketplace -name executor.env.sample | sort -V | tail -1)
# 방식 A (전역):
mkdir -p ~/.unskein && cp "$TPL" ~/.unskein/executor.env && chmod 600 ~/.unskein/executor.env
# 방식 B (프로젝트마다 — 프로젝트 작업폴더를 먼저 만들고 그 안에 .unskein):
PROJ=~/webapp                                          # ← 이 실행기 전용 작업폴더(이름 자유). 이 줄이 없으면 ~/.unskein(전역)이 돼버린다
mkdir -p "$PROJ/.unskein/creds" "$PROJ/.unskein/work"
cp "$TPL" "$PROJ/.unskein/executor.env" && chmod 600 "$PROJ/.unskein/executor.env"
```
```shell
# executor.env 핵심 값 (편집기로):
export UNSKEIN_HOME=/home/<user>/<프로젝트폴더>/.unskein # ← 방식 B 만, 절대경로, 파일 맨 위
export UNSKEIN_API=https://unskein.mupai.studio
export UNSKEIN_MORI_TOKEN=<복사한 토큰>
export UNSKEIN_WATCH_BUSINESS="ACME Corp"               # 공백 값 따옴표!
export UNSKEIN_WATCH_PROJECT="WebApp"                   # 방식 A 는 이 줄을 비움(비즈니스 전체)
#export UNSKEIN_MAX_CONCURRENCY=2                       # 동시 다오 수(기본 3 — 머신 여유 보고)
```
4. **확인**:
```shell
source <경로>/executor.env
curl -s "$UNSKEIN_API/api/health"      # {"status":"ok"}
curl -s -o /dev/null -w '%{http_code}\n' -H "X-Mori-Token: $UNSKEIN_MORI_TOKEN" "$UNSKEIN_API/api/mori/scope"   # 200
```
- `401` = 토큰 오타 / kind 가 planner / 다른 서버 토큰 섞임.

> ⚠️ **하지 말 것**: ① **다른 머신·프로젝트의 executor.env/토큰 복사** — 토큰 공유 = 서버 펜싱 전멸(§8 ②). ② 명령줄에 토큰 타이핑. ③ 공백 값 따옴표 누락. ④ **방식 B 에서 bashrc 자동 로드** — 전역 env 가 매 셸에 깔리면 프로젝트 env 에 없는 변수(git 토큰·WATCH·정책)가 전역 값으로 남는 **레이어링 누출**. 자동 로드는 방식 A 전용 옵션, 방식 B 는 세션마다 명시 `source`. 기존 bashrc 에 자동 로드 줄이 있으면 **제거**.
> 두 타겟(서버가 둘일 때)을 오갈 땐 `executor.env` / `executor.local.env` 처럼 파일만 나눠 **그 하나만** source(짝 섞이면 401).

### 4.2 git 클론 자격증명 — 멀티프로젝트의 급소

다오의 클론에 gh 폴백은 **없다** — 자격증명이 없으면 그 작업은 QUESTION 회수. 자격증명은 `$UNSKEIN_HOME/creds`(방식 A 는 `~/.unskein/creds`) 또는 env.

> 🔑 **HTTPS 냐 SSH 냐는 서버의 `repo_url` 형식이 정한다**(내 키/토큰이 아니라): `https://…` → 토큰(`UNSKEIN_GIT_TOKEN`), `git@…`(예: `git@github.com:ACME/WebApp.git`)/`ssh://…` → SSH 키.

- **A. HTTPS**: 편집기로 `<creds>/.env` 에 `UNSKEIN_GIT_TOKEN=github_pat_xxx`(권한 600, 폴더 700). 호스트 여럿이면 `UNSKEIN_GIT_TOKEN_GITHUB_COM` 등. **주의 — 해석 우선순위**: 셸 env 가 `creds/.env` 보다 **먼저** — env 에 옛 값이 남으면 creds 를 고쳐도 옛 토큰이 쓰인다.
- **B. SSH**: 개인키를 `<creds>/id_ed25519`(600)에 — 자동 인식(차선 `id_rsa`). 다른 이름이면 executor.env 에 `UNSKEIN_SSH_KEY=<전체경로>`. 공개키를 GitHub 에 등록(계정 키 또는 deploy key). 키 권한·known_hosts 는 실행기가 자동 보정.
- 🔑 **커버리지 규칙**: 자격증명 해석은 **호스트 단위**. **그 세션 스코프에 들어올 모든 repo 를 자격증명 하나가 커버**해야 한다 — 같은 org 면 fine-grained PAT 하나로 충분, org 가 흩어지면 classic PAT 또는 **방식 B 로 프로젝트별 분리**(B 의 장점 — creds 가 프로젝트별이라 org 마다 다른 PAT 가능). 프로젝트 추가 시마다 재점검.
- **실전 확인은 첫 클론** — `unskein-setup` S2 가 실클론으로 검증한다. `/unskein:status` 의 "creds ✅" 는 폴더 존재만.
- **교체/회전**: `unskein-setup` 재실행(자격증명 갱신) — 옛 토큰·공개키는 발급처에서 폐기.

### 4.3 preflight 게이트 (S3)

`/unskein:status` 가 작업 잡기 전 게이트(preflight)를 그대로 보여준다 — 다오 CLI·git·gh 인증·dao-skills·creds·work·서버 도달 중 하나라도 미충족이면 **폴링 시작 안 함**. v1.18.0 부터 추가로:
- **`[경로]` 줄 2개** — 이 세션의 유효 `UNSKEIN_HOME`·creds·work 절대경로 표시. **방식 B 검수 필수**: 프로젝트 경로가 맞는지 눈으로 확인(잘못된 홈으로 붙은 watch 는 이 줄 없이는 무증상).
- **`상태 루트 정합(UNSKEIN_HOME)`** — `UNSKEIN_HOME` 을 설정했는데 개별 변수(`UNSKEIN_CRED_DIR`/`UNSKEIN_WORK_ROOT`, 보통 전역 env 잔재)가 creds/work 를 홈 밖으로 끌고 가면 **치명 실패로 차단**(상태 반쪽 분산 사고 방지). SSH 키 경로(`UNSKEIN_SSH_KEY`)가 홈 밖이면 `[경고]`(의도한 외부 키면 무시 — 아니면 다른 신원 키 누출). 복구: 그 변수·bashrc 자동 로드 줄 제거 후 프로젝트 env 만 다시 source.
- **`[경로] env 파일(cwd 폴백 로드)`**(v1.20.0+) — source 를 안 해서 `run_once` 가 cwd 폴백으로 `executor.env` 를 자동 로드했으면 **어느 파일에서 왔는지** 표시. 프로젝트가 맞는지 눈으로 확인(엉뚱한 조상 `.unskein` 을 주웠나).
- **`전역 ~/.unskein 무력화(멀티프로젝트)`**(v1.20.0+) — 프로젝트 격리 모드인데 전역 `~/.unskein` 이 남아 있으면 `[경고]`. cwd 폴백이 조상 전역을 주울 수 있어서다. 복구: `mv ~/.unskein ~/.unskein_back`(§7).

## 5. 기동 (방식별)

### 방식 A — 한 세션
```shell
cd ~/unskein-executor            # 홈베이스(빈 폴더, 세션 히스토리 정리용 — 없으면 mkdir)
source ~/.unskein/executor.env
claude
```
```
/unskein:status                                     # preflight + watch 대상 검증 (~20초)
/unskein:run bis "ACME Corp" prj "WebApp"           # 첫 가동: 1건 사람 입회
/unskein:watch                                      # 검증 후 자율 루프 (bis 만 env 에 — 비즈니스 전체)
```

### 방식 B — 프로젝트마다 한 세션 (터미널 탭/tmux 창 하나씩)
```shell
# 터미널 1
cd /home/<user>/project01 && source ./.unskein/executor.env && claude
#  → /unskein:status  →  /unskein:watch     (env 의 bis+prj 가 이 프로젝트로 고정)
# 터미널 2
cd /home/<user>/project02 && source ./.unskein/executor.env && claude
#  → /unskein:status  →  /unskein:watch
```
- executor.env 맨 위의 `UNSKEIN_HOME` 절대경로 덕에 **어느 cwd 에서 source 해도** 그 프로젝트 상태로 확정되지만, 프로젝트 디렉토리에서 여는 걸 관례로(세션 히스토리도 프로젝트별로 갈린다).
- 두 세션의 work·creds 는 서로 완전 격리 — `project01/.unskein/work` 와 `project02/.unskein/work`.

### 공통
- ⚠️ **단발 `/unskein:run` 은 범위 차단이 없다** — 인자 없이 돌리면 멤버십 전체에서 아무거나 선점. **항상 `bis`/`prj` 인자 지정.** (watch 는 원격+무범위면 시작 거부 — `UNSKEIN_ALLOW_UNSCOPED=1` 은 비권장.)
- env 를 고쳤으면 claude 를 껐다 다시(세션은 시작 시점 env 고정). **플러그인 업데이트 후에도 watch 재시작 필수**(장수 루프는 import 시점 상수 고정 — preflight 의 "플러그인 최신" 경고는 비차단).
- **source 우선·cwd 폴백**(v1.20.0+): `source <프로젝트>/.unskein/executor.env` 를 권장하되, 깜빡해도 그 `.unskein` 아래에서 띄우면 자동 로드된다. ⚠️ **프로젝트마다 새 셸** — 이전 프로젝트의 `UNSKEIN_MORI_TOKEN` 이 남으면 source 우선상 그게 이겨 엉뚱한 프로젝트로 붙는다(실측된 사고 유형). 헷갈리면 `env -u UNSKEIN_MORI_TOKEN …` 로 비우고 시작.
- **빈 tick 은 정상** — claim 가능 상태는 `plan`/`exec`/`test`/`inspect`/`answered`. `backlog` 는 웹 칸반에서 `plan` 으로 올려야 선점(진행 승인 게이트). `test` 는 기본으로 EXECUTOR 가 안 집는다(TESTER 게이트, 설계 결정 ADR-0014) — 자율 루프로 잇고 싶으면 `UNSKEIN_AUTO_ADVANCE_TEST=1`(설계 결정 ADR-0016, 프로덕션은 끔).
- **같은 카드 여러 번 선점 = 정상** — 단계(구현→검증→마감)마다 재선점, 폴더는 단계 간 보존.
- 셸에서 직접(claude 밖): `BIN=$(dirname "$(find ~/.claude/plugins/cache/unskein-marketplace -name unskein-loop -path '*/bin/*' | sort -V | tail -1)")` → `"$BIN/unskein-once"` / `"$BIN/unskein-loop" 30 0`.

### 성공 가동의 로그 모양 (예시)
```
[preflight] 작업 전 준비 점검:  (…항목 OK, [경로] 상태 루트 표시)
[loop] 시작 — interval=30s concurrency=3 api=… watch=…
[tick 1] claim task#101 '…' repo=…/WebApp.git (진행 중 1/3)   ← 프로젝트 A
[tick 2] claim task#102 '…' repo=…/WebAPI.git (진행 중 2/3)   ← 프로젝트 B, 동시(방식 A)
[dao] claude -p 실행 (cwd=…/work/101, timeout=600s) … returncode=0 · session_id=…
[tick 7] claim task#101 …                                     ← 같은 카드 재선점 = 다음 단계(정상)
```
- 실패 신호: `QUESTION:`(막힘 — 웹에서 답변) / `claim 실패`(연결·토큰) / `is_error`·timeout(턴 실패 → 질문 회수).
- 완주 판정: 카드 `done` + 대상 repo 에 feature 브랜치 PR(`gh pr list --repo <repo>`).
- 멀티 확인: 방식 A 는 한 로그에 `진행 중 2/N`, 방식 B 는 터미널 2개가 각자 자기 프로젝트만.

## 7. 이행 — 기존 전역 실행기 → 방식 B

1. **in-flight 0 확인**: 그 프로젝트에 waiting/answered·단계 중간(plan~inspect 사이) 작업이 없을 때만 이행 — 대화턴 resume 은 작업 폴더 경로에 묶여 있어 홈 전환이 세션 이어받기를 끊는다. 남아 있으면 완주시키고 이행.
2. bashrc 의 전역 자동 로드 줄 **제거**, 새 셸에서 `env | grep UNSKEIN_` 이 **깨끗한지** 확인(전역 잔존 변수 = 레이어링 누출·split 원인).
3. 프로젝트 디렉토리에서 `unskein-setup` 실행(S0 에서 `$PWD/.unskein` 확정, **토큰은 새로 발급**).
4. `/unskein:status` 로 `[경로]` 줄이 프로젝트 경로인지 + `상태 루트 정합` OK 확인.
5. **전역 `~/.unskein` 무력화** — `mv ~/.unskein ~/.unskein_back`(개명, 되돌리기 가능). ⚠️ v1.20.0 부터 cwd 폴백이 **조상인 전역 `~/.unskein` 을 주울 수 있어**, 그냥 남겨두면 프로젝트 세션이 전역으로 조용히 붙을 수 있다(더는 "무해"하지 않다). preflight 의 `전역 ~/.unskein 무력화` 경고가 이를 알린다. bashrc 자동 로드 줄도 반드시 제거.

## 8. 실행기 ×N 안전 규칙 (방식 무관 불변)

시스템이 강제하는 건 "원격+무범위 watch 차단"과 "task 단위 원자적 이중 선점 방지"뿐 — 나머지는 규칙:

| 규칙 | 위반 시 (검증으로 확정된 사고 경로) |
|---|---|
| **① 스코프 서로소** — 세션마다 bis/prj 안 겹치게 | claim 후보에 `executed_by` 필터가 없어(affinity 부재) A 가 하던 작업의 **다음 단계**를 B 가 정상 타이밍에 선점 → 빈 트리 검증·resume 이어받기 실패 |
| **② 세션당 전용 토큰** — 모리 토큰 공유 금지 | 펜싱이 `token.id` 비교라 공유 = **펜싱 전멸**(heartbeat 단절 후 좀비 쓰기 통과) |
| **③ 상태 루트 분리** — 같은 distro 다중 세션이면 `UNSKEIN_HOME` 프로젝트별(v1.18.0), distro 격리면 자동 충족 | 같은 `WORK_ROOT/<task_id>` 를 두 프로세스가 만지거나 GC 가 남의 waiting 폴더 삭제. v1.18.0 preflight 가 부분 분산(split)은 차단하지만 **완전 동일 HOME 두 세션**은 못 잡는다 — 규칙으로 |
| ④ 서버 DB 는 PostgreSQL 전제 | 이중 선점 1차 방어(`SKIP LOCKED`) 실효 조건 |

권장: **한 프로젝트 = 한 watch 세션 = 전용 토큰 + 전용 `UNSKEIN_HOME`**(방식 B) 또는 **한 비즈니스 = 한 세션**(방식 A). 근거: 설계 결정 ADR-0020.

## 9. 막히면

> 스킬은 셸 명령이 아니라 **claude 세션 대화에서 이름으로 호출**("unskein-doctor 로 진단해줘").

- 환경·연결 진단 → `unskein-doctor`(preflight 단일출처: 증상→원인→복구→재검증).
- **401** → API·토큰 짝(다른 서버 토큰) / kind(planner 를 mori 자리에).
- **빈 tick 만** → claim 가능 상태 작업 없음(웹 칸반에서 `plan` 으로) / `test` 에 몰려 있으면 TESTER 게이트(§5 공통).
- **특정 프로젝트만 QUESTION 회수** → §4.2 커버리지(그 repo 권한) + 셸 env `UNSKEIN_GIT_TOKEN` 잔존(creds 보다 우선).
- **`상태 루트 정합` 실패** → 전역 개별 변수·bashrc 자동 로드 잔재(§4.3, §7-2).
- **엉뚱한 프로젝트 작업을 잡음** → 그 세션 env 의 `UNSKEIN_WATCH_*` 확인 + `[경로]` 줄로 어느 홈인지 확인(방식 B 에서 다른 프로젝트 env 를 source 한 사고).
- `source` 오류 → 공백 값 따옴표.
- 빌드 검증 실패 QUESTION → 대상 스택 런타임이 **WSL 안에** 있는지(§2).
- 화면 검증 → `unskein-test`(CDP) — 헤드리스 WSL 다오는 브라우저를 못 띄우므로 **윈도우에서**(기본 9222). 테스터 세션을 여러 개 돌리면 **세션마다 포트를 갈라**(`start.ps1 -Port <n>` + `remote.js --port/CDP_PORT`, v1.19.0) — 포트↔프로필 1:1 이라 로그인(JWT)이 완전 분리된다. TESTER 설치는 같은 폴더의 `테스터설치_단일프로젝트.md`.
