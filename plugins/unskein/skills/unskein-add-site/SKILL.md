# UnSkein — 사이트(프로젝트) 추가·자격증명 관리

먼저 `unskein-connect` 로 서버에 연결돼 있어야 합니다.

모리 클라이언트가 새 프로젝트를 개발할 수 있도록, 그 프로젝트를 클라이언트에 등록하고 **등록 시점에 즉시 처리할 수 있도록 프로비저닝**합니다(저장소 실클론 + 다오 스킬 + 런타임 확인). 등록 이후 자격증명을 교체·추가해야 할 때도 같은 스킬로 처리합니다. 클라이언트 주인(사용자)과 대화하며 아래 순서로 진행합니다. 값이 빠지면 임의로 채우지 말고 사용자에게 물어 멈춥니다.

## 1. 프로젝트 성격 파악

- 사용자에게 프로젝트 이름과 저장소 주소를 묻습니다.
- 접근 가능하면 저장소를 얕게 살펴보고, 아니면 사용자에게 물어 언어·스택을 파악합니다 (node / python / 기타).
- 다오가 그 프로젝트를 빌드·테스트하려면 필요한 런타임이 WSL에 있어야 합니다. 없으면 설치를 안내합니다.

## 2. git 정보·자격증명 받기

- 저장소 주소를 받고 전송 방식을 판정합니다: `https://` → 토큰, `git@`/`ssh://` → SSH 개인키.
- 자격증명은 **`unskein-connect` 가 만든 단일 env 파일(`~/.unskein/env.<타겟>`)** 에 둡니다(서버 토큰과 한 파일):
  - HTTPS: 개인 액세스 토큰을 **env 파일의 `UNSKEIN_GIT_TOKEN=`** 에 둡니다(오케스트레이터가 env 를 `creds/.env` 보다 우선하므로 별도 `creds/.env` 는 두지 않습니다 — 있으면 지워 혼선 방지). `gh pr create` 용 **`GH_TOKEN=`** 도 같은 PAT 로 채우면 `gh auth login` 을 생략할 수 있습니다. 호스트가 여럿이면 `UNSKEIN_GIT_TOKEN_<HOST>=` 로 구분합니다.
  - SSH: 개인키는 파일이라 `UNSKEIN_CRED_DIR`(기본 `~/.unskein/creds`)의 `id_ed25519`(또는 `id_rsa`)에 두고 권한을 `600` 으로 맞춥니다. 커스텀 이름이면 env 파일에 `UNSKEIN_SSH_KEY=<경로>`(경로만 — 키 내용은 파일).
- 토큰·키는 화면에 다시 출력하지 않습니다. 저장소 주소나 git 설정에 토큰을 넣지 않습니다.
- 토큰 값은 편집기로 env 파일에 직접 입력하도록 안내합니다. 명령줄 인자로 입력하면(예: `echo` 로 추가) 셸 기록에 값이 남으므로 그렇게 하지 않습니다.

## 3. 프로비저닝·검증 — 등록 시점에 "즉시 처리 가능" 을 보장

단순 접근 확인(`ls-remote`)을 넘어, 이 클라이언트가 이 프로젝트를 **바로 처리할 수 있는 상태**로 갖춥니다. 작업 때 가서야 환경 문제로 실패하는 것을 설정 단계에서 선제 차단합니다. (역할 분담: **add-site = 1회 무거운 준비**, **모리 preflight = 매 작업 전 가벼운 점검**. add-site 가 갖춰 두면 preflight 가 통과합니다.)

1. **저장소 실클론**: `git ls-remote` 로 그치지 말고, 받은 자격증명으로 작업 루트(`UNSKEIN_WORK_ROOT`, 기본 `~/.unskein/work`)에 **실제로 클론**합니다(`$UNSKEIN_WORK_ROOT/<repo>`). 토큰·키가 end-to-end 로 동작하는지 확인됩니다. (이후 작업 때 모리의 `prepare_repo` 가 이 클론을 fetch·리셋으로 재사용합니다.)
2. **다오 스킬 심기 확인**: 다오 스킬 원본(plugin `dao-skills/`, 또는 `UNSKEIN_DAO_SKILLS`)을 작업 루트로 복사해 `WORK_ROOT/CLAUDE.md` + `WORK_ROOT/.claude/skills/` 가 깔리는지 확인합니다. (작업 때 모리가 매번 다시 심지만, 여기서 원본·경로가 맞는지 미리 검증.)
3. **빌드 런타임 확인**: 1번에서 파악한 스택의 런타임이 WSL 에 **실제 존재**하는지 `which` 등으로 확인합니다(node/python 등). 안내만 말고 실제로 검사하고, 없으면 설치를 안내한 뒤 멈춥니다.
4. 모두 통과하면 "이 프로젝트로 작업 큐의 작업을 **즉시 처리할 준비 완료**"를 알리고, `/unskein:run` 또는 `/unskein:watch` 를 안내합니다.
- 어느 단계든 실패하면 사유(인증·네트워크·주소 형식·런타임 부재)를 그대로 보여주고 멈춥니다. 다른 값으로 우회하지 않습니다(fallback 금지).

## 4. 자격증명 갱신·관리

이미 등록된 프로젝트의 토큰·키를 교체하거나 호스트를 추가할 때 사용합니다. 모든 갱신 작업에서 비밀 값은 화면에 출력하지 않고, 저장소 주소·git 설정에 토큰을 넣지 않습니다. 값이 빠지면 임의로 채우거나 우회하지 말고 사용자에게 물어 멈춥니다(fallback 금지).

| 작업 | 대상 | 절차 |
|------|------|------|
| 토큰 회전 | env 파일의 `UNSKEIN_GIT_TOKEN=`(·`GH_TOKEN=`) | 값을 새 토큰으로 교체한다. 호스트가 여럿이면 해당 `UNSKEIN_GIT_TOKEN_<HOST>=` 줄만 교체한다. |
| 호스트 추가 | env 파일 | 추가 호스트의 토큰을 `UNSKEIN_GIT_TOKEN_<HOST>=` 로 새 줄에 둔다. 기존 호스트 값은 그대로 둔다. |
| SSH 키 교체 | `creds/id_ed25519`(또는 `id_rsa`) | 새 개인키 파일로 교체하고 권한을 `600` 으로 맞춘다. |

### 토큰 회전

- env 파일(`~/.unskein/env.<타겟>`)의 `UNSKEIN_GIT_TOKEN=`(또는 `UNSKEIN_GIT_TOKEN_<HOST>=`, 필요 시 `GH_TOKEN=`) 값을 편집기로 새 토큰으로 교체합니다. 명령줄 인자로 입력하지 않습니다(셸 기록에 값이 남습니다).
- 교체 후 옛 토큰은 발급처(예: GitHub)에서 폐기하도록 사용자에게 안내합니다. 폐기하지 않으면 노출된 토큰이 계속 살아 있습니다.
- 토큰 값은 화면에 출력하지 않습니다. 설정 여부만 "설정됨 / 없음" 으로 알립니다.

### 호스트 추가

- 저장소 호스트가 여럿이면 호스트별 토큰을 `UNSKEIN_GIT_TOKEN_<HOST>=` 로 구분합니다 (예: `UNSKEIN_GIT_TOKEN_GITHUB_COM=`, `UNSKEIN_GIT_TOKEN_GITLAB_COM=`).
- 기존 호스트의 토큰은 그대로 두고, 새 호스트 줄만 추가합니다.

### SSH 키 교체

- `creds/id_ed25519`(또는 `id_rsa`)를 새 개인키로 교체합니다.
- 교체 후 권한을 `600` 으로 맞춥니다 (`chmod 600`).

### 갱신 후 재검증

- 교체·추가한 자격증명이 실제로 동작하는지 확인합니다 — HTTPS 토큰은 오케스트레이터 askpass 경로로만 쓰이므로 `/unskein:run`(실클론) 또는 §3 프로비저닝으로 확정합니다(맨손 `git ls-remote` 는 env 토큰을 안 탈 수 있음). SSH 키는 `source ~/.unskein/env.<타겟>` 후 `git ls-remote <repo>` 로 확인.
- 성공하면 갱신 완료를 알립니다. 토큰을 회전한 경우 옛 토큰 폐기 안내를 다시 확인합니다.
- 실패하면 사유(인증·네트워크·주소 형식)를 그대로 보여주고 멈춥니다. 다른 값으로 우회하지 않습니다.
