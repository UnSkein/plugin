# 로지컬 씽킹 (logical-thinking)

MECE 원칙으로 복잡한 주제를 체계적으로 다루는 통합 도구다. 분해 → 검증 → 변환(폴더/표) → 메모리 정리까지 다섯 가지 작업을 하나의 스킬로 제공한다.

작업마다 산출물과 절차가 다르므로, 먼저 아래 라우팅 표에서 지금 요청이 어떤 작업인지 정한 뒤 해당 작업 절로 이동한다. 모든 작업은 공통으로 아래 "공통 원칙"을 기반으로 하므로, 어떤 작업을 하든 공통 원칙을 먼저 읽는다.

> 이 스킬은 외부 의존 없이 이 폴더만으로 작동한다. 다른 Claude Code 환경으로 옮기려면 `logical-thinking/` 폴더를 그대로 복사하면 된다.

## 작업 라우팅

| 사용자 요청 | 작업 | 산출물 |
|------------|------|--------|
| 복잡한 주제를 MECE로 분해, 번호체계 만들기 | **A. 분해** | `thinking_result_[주제].md` |
| 분해 결과가 MECE를 지키는지 품질 검증 | **B. 검증** | `verification_report_[주제].md` |
| 분해 결과를 실행 가능한 폴더구조로 변환 | **C. 폴더 변환** | `small_pictures/` 폴더 |
| 분해 결과를 엑셀용 계층 CSV로 변환 | **D. CSV 변환** | `logical_tree_[주제].csv` |
| 메모리(MEMORY.md + 상세 파일) 구조 정리 | **E. 메모리 정리** | 메모리 재구성 리포트 |

요청 키워드를 보고 해당 작업 절로 이동해 그 절차를 따른다. A에서 만든 결과를 B/C/D가 입력으로 받는 흐름이고, E는 같은 원칙을 메모리 구조에 적용하는 독립 작업이다.

---

## 공통 원칙 (모든 작업의 기반)

### 1. MECE (Mutually Exclusive, Collectively Exhaustive)
- **ME (상호배타)**: 각 항목이 서로 겹치지 않는다.
- **CE (완전망라)**: 필요한 요소를 빠뜨림 없이 모두 포함한다.

### 2. 동일 기준 형제 브랜치
같은 부모 아래 모든 형제는 반드시 동일한 분해 기준을 사용한다.

```
올바름: 인프라(도메인별) / 데이터(도메인별) / 실행(도메인별)
잘못됨: 1단계(시간순) / DB(기술스택) / 높은우선순위(우선순위)
```

### 3. 집합 분류
- **닫힌 집합**: 논리적으로 완전히 정의 가능 → "기타" 불필요
- **열린 집합**: 확장 가능 → "기타" 항목 필수

## 작업 A: 주제 분해 (think)

복잡한 주제를 MECE 원칙으로 분해하여 번호체계 문서를 생성한다.

### 옵션
- `--depth shallow|medium|deep` (기본: medium)
- `--criteria 분해기준` (시간축, 도메인, 우선순위, 기능, 리스크)

### 실행 절차

1. **최상위 목표 설정** — 구체적이고 측정 가능하게.
2. **1차 분해 기준 선택** — 아래 중 주제에 가장 맞는 것을 고른다.
   ```yaml
   시간축: "현재 → 단기 → 장기"
   도메인: "Frontend → Backend → Database"
   우선순위: "필수 → 중요 → 선택"
   기능: "입력 → 처리 → 출력"
   리스크: "안전 → 위험 → 실험"
   ```
3. **브랜치 생성** — 동일 기준으로 형제를 만든다 (공통 원칙 2 참조).
4. **MECE 검증** (공통 원칙 1 참조)
   - ME: "각 브랜치가 겹치는가?" / "하나의 항목이 여러 곳에 속하는가?"
   - CE: "모든 경우를 다뤘는가?" / "빠진 영역이 있는가?"
5. **하위 분해** — 원자적 Task(1~3일 완료)까지 내려간다.
6. **문서 출력**

### 출력 템플릿

파일명: `thinking_result_[주제명].md`

```markdown
# [주제명] 로지컬 씽킹 결과

## 최상위 목표
**[구체적이고 측정 가능한 목표]**

## 1차 분해 ([선택된 기준])

### 1. [브랜치명]
- **1.1 [하위 브랜치]**
  - 1.1.1 [구체적 Task]
  - 1.1.2 [구체적 Task]

### 2. [브랜치명]
- **2.1 [하위 브랜치]**
  - 2.1.1 [구체적 Task]

### n. 기타 (열린 집합인 경우)

## MECE 검증 결과
- **ME 확인**: [중복 여부]
- **CE 확인**: [누락 여부]
- **집합 유형**: [닫힌 | 열린]
- **분해 기준 일관성**: [준수 여부]

## 품질 체크리스트
- [ ] 모든 형제 브랜치가 동일 기준으로 분해됨
- [ ] 브랜치 간 겹치는 내용 없음
- [ ] 누락된 중요 영역 없음
- [ ] 열린 집합에 "기타" 항목 존재
- [ ] 각 브랜치가 상위 목표에 직접 기여
- [ ] 원자적 Task 수준 (1~3일 완료 가능)
```

## 작업 B: MECE 검증 (verification)

분해 결과물을 MECE 원칙과 품질 기준으로 검증하여 리포트를 생성한다.

### 옵션
- `--focus me|ce|structure|quality` (기본: 전체)
- `--엄격` (엄격 모드)
- `--report detailed|summary` (기본: detailed)

### 검증 5단계

**1단계: 구조 분석**
- 최상위 목표의 명확성
- 깊이별 분해의 적절성
- 브랜치 간 논리적 관계

**2단계: ME (상호배타성) 검증** — 공통 원칙 1 참조
```yaml
질문:
  - "각 브랜치가 서로 겹치는가?"
  - "하나의 항목이 여러 브랜치에 속할 수 있는가?"
  - "분해 기준이 형제 브랜치에서 일관되는가?"
위반 표시:
  - (중복id01), (중복id02): 동일 내용
  - 기준 불일치: 분해 기준이 혼재된 지점
통과 조건:
  - 항목이 둘 이상의 브랜치에 속하지 않음
  - 브랜치 경계가 명확
  - 동일 분해 기준이 일관 적용됨
```

**3단계: CE (완전망라성) 검증** — 공통 원칙 1 참조
```yaml
질문:
  - "모든 경우를 다뤘는가?"
  - "빠진 영역이 있는가?"
  - "새로운 하위 항목 가능성은?"
누락 표시:
  - (누락): 빠진 내용
  - 집합 유형 판단: 닫힌 vs 열린
  - 열린 집합에서 "기타" 브랜치 필요성
통과 조건:
  - 모든 영역 포함
  - 닫힌 집합: 논리적 완전 정의
  - 열린 집합: "기타"로 확장성 보장
```

**4단계: 품질 기준 평가**
```yaml
구조 품질:
  - 동일 기준 원칙 준수
  - 중복 없음
  - 누락 없음
  - 확장성
내용 품질:
  - 상위 목표 기여도
  - 실행 가능성
  - 완료 조건 명확성
실용성:
  - 팀 이해도
  - 관리 용이성
  - 구조 유지성
```

**5단계: 원자적 Task 검증**
- 1~3일 안에 완료 가능한 크기인가?
- 독립적으로 실행 가능한가?
- 성공/실패 판단 기준이 있는가?

### 점수 체계
```yaml
A (90-100): MECE 완벽, 즉시 활용 가능
B (80-89): 대부분 충족, 경미한 개선
C (70-79): 기본 충족, 중간 개선 필요
D (60-69): 일부 미충족, 상당한 개선
F (<60): 대부분 미충족, 재작업 필요

필수 통과: ME 위반 0개, CE 누락 0개, 총점 80점 이상
```

### 리포트 템플릿

파일명: `verification_report_[주제명].md`

```markdown
# [주제명] 검증 리포트

## 개요
- 대상: [파일명]
- 전체 평가: [A-F]

## 구조 분석
- 목표 명확성: [평가]
- 분해 기준: [기준명] — [평가]

## ME (상호배타성) — [통과/위반]
### 위반사항
- [중복 항목]: [내용] → [위치]
### 개선안
- [방법]

## CE (완전망라성) — [통과/위반]
### 누락사항
- [누락 영역]: [내용] → [추가 위치]
### 개선안
- [방법]

## 품질 점수
- 구조: __/100
- 내용: __/100
- 실용성: __/100

## 개선 로드맵
### 우선순위 1 (즉시)
- 문제 → 영향도 → 해결방안
### 우선순위 2 (단기)
- 문제 → 해결방안

## 재검증 체크리스트
- [ ] ME 위반 해결
- [ ] CE 누락 보완
- [ ] 품질 80점 이상 달성
- [ ] 원자적 Task 기준 충족
```

## 작업 C: 실행 폴더구조 변환 (tree)

번호체계 분해 결과를 실행 가능한 폴더구조 + Module Process 파일로 변환한다.

### 옵션
- `--workspace [경로]` (기본: sessions/current)
- `--extract atomic_tasks` (원자적 Task만 추출)

### 실행 절차

1. **분석** — 번호체계 문서를 파싱하고 계층구조를 이해한다.
2. **추출** — 원자적 Task를 식별한다 (1~3일 완료, 독립 실행 가능, 성공/실패 판단 가능).
3. **구조화** — 폴더 + Module Process 파일을 생성한다.
4. **검증** — 폴더 완전성을 확인한다.

### 생성 구조

```
[workspace]/small_pictures/
├── [실행단위1_snake_case]/
│   ├── scope_[단위명].md
│   ├── status_analysis_[단위명].md
│   ├── final_design_[단위명].md
│   └── execution_[단위명].md
├── [실행단위2_snake_case]/
│   └── (동일 4파일)
└── workset_controller/
    └── (동일 4파일)
```

## 작업 D: 엑셀 CSV 변환 (csv)

번호체계 분해 결과를 엑셀에 최적화된 계층 CSV로 변환한다.

### 옵션
- `--columns basic|extended|full` (기본: extended)
- `--format vertical|sitemap` (기본: vertical)
- `--status-init pending|ready` (기본: pending)

### Format 1: Vertical Column (기본)

계층을 Level1~4 컬럼에 배치하고, 빈 셀로 트리 구조를 표현한다.

**컬럼 구조**
```
basic:    Level1,Level2,Level3,Level4,Status
extended: Level1,Level2,Level3,Level4,Status,Duration,Priority,Owner
full:     Level1,Level2,Level3,Level4,Status,Duration,Priority,Owner,Dependencies,Start_Date,End_Date,Completion,Risk_Level,Notes
```

**배치 규칙**
- Level1: 최상위만 입력, 나머지 행은 빈 셀
- Level2: Level1이 빈 행에만 입력
- Level3: Level1,2가 빈 행에만 입력
- Level4: Level1,2,3이 빈 행에만 입력

**예시**
```csv
Level1,Level2,Level3,Level4,Status,Duration,Priority,Owner
Epic Framework MVP,,,Planning,375분,P0,Team
,데이터 영역,,,In Progress,165분,P0,Backend
,,API 개발,,Pending,60분,P0,Dev1
,,,RESTful 엔드포인트,Pending,30분,P0,Dev1
,,,GraphQL 스키마,Pending,30분,P1,Dev1
,,DB 스키마,,Completed,45분,P0,Dev2
,실행 영역,,,Pending,90분,P1,Frontend
```

### Format 2: Sitemap (`--format sitemap`)

Sitemap/UI 구조용. 번호 체계 없이 순수 계층만 표현한다.

**핵심 규칙**
- 제목 위치 = 계층 깊이 (1뎁스 → 컬럼1, 2뎁스 → 컬럼2, ... 5뎁스 → 컬럼5)
- 설명 위치 = 항상 컬럼12 (고정)
- Padding commas = `11 - depth`

**예시**
```csv
제목,,,,,,,,,,,설명,타입,상태,우선순위
Epic Framework,,,,,,,,,,,Complete low-code form builder,시스템,완료,P0
,Authentication,,,,,,,,,,사용자 인증 시스템,기능모듈,완료,P0
,,Login Page,,,,,,,,,로그인 페이지,페이지,완료,P0
,,,LoginForm,,,,,,,,이메일/비밀번호 입력 폼,컴포넌트,완료,P0
```

### Status 값
```yaml
Planning: 계획 단계
Pending: 대기 중
Ready: 시작 준비됨
In Progress: 진행 중
Review: 검토 중
Completed: 완료
Blocked: 차단됨
On Hold: 보류
Cancelled: 취소
```

### Priority 값
```yaml
P0: Critical (블로킹, 즉시 필요)
P1: High (중요, 대기 가능)
P2: Medium (일반)
P3: Low (여유 시)
```

## 작업 E: 메모리 구조 정리 (memory)

MECE 원칙으로 Claude Code의 auto memory 파일들(MEMORY.md + 상세 파일)의 구조를 검증하고 최적화한다.

### 메모리 디렉토리 확인 (먼저 할 것)

메모리 경로는 환경마다 다르므로, 시작 전에 현재 환경의 메모리 디렉토리를 먼저 확인한다.

- Claude Code 메모리는 보통 `~/.claude/projects/<현재 프로젝트 경로를 인코딩한 폴더>/memory/`에 있고, 그 안에 `MEMORY.md`가 들어 있다.
- 세션의 시스템 안내에 메모리 경로가 명시돼 있으면 그것을 사용한다. 없으면 `ls ~/.claude/projects/*/memory/MEMORY.md`로 실제 위치를 확인한 뒤, 현재 작업 중인 프로젝트에 해당하는 디렉토리를 대상으로 삼는다.

확인된 디렉토리의 구조 예:
```
<메모리 디렉토리>/
├── MEMORY.md          # 목차 + 핵심 규칙 (200줄 한정, 항상 로드)
├── details.md         # 상세 참조 (앵커 링크)
└── [topic].md         # 주제별 분리 파일 (필요시 추가)
```

### 3계층 원칙

**Layer 1: MEMORY.md (항상 로드, 200줄 한정)**
```yaml
역할: 포인터 인덱스 (내용 인라인 금지)
포함:
  - 상세 파일 포인터 (> 링크 + 1줄 요약)
  - 토픽 파일이 없는 독립 규칙만 예외적 인라인 허용
금지:
  - 코드 스니펫
  - 구현 디테일
  - 히스토리/날짜 (참조용 날짜 1개는 허용)
  - 스킬에 이미 있는 내용 (중복 금지)
  - 코드에서 파생 가능한 패턴 (grep/ls로 확인 가능한 것)
```

**Layer 2: 주제별 파일 (필요시 로드)**
```yaml
역할: 특정 도메인의 상세 지식
분할_기준: MECE — 하나의 주제가 하나의 파일
네이밍: kebab-case, 20자 이내 (타입 접두사 불필요 — 프론트매터에 type 있음)
크기: 200줄 이내 권장
포함:
  - "코드만 보면 실수할 수 있는" gotcha/경험
  - 의사결정 이력
  - 구현 패턴 상세
  - 테이블/비교표
금지:
  - 스킬에 이미 있는 절차/방법
  - 코드에서 파생 가능한 구조/경로/시그니처
업데이트_규칙:
  - 새 지식 발생 시 기존 토픽 파일이 있으면 해당 파일에 추가 (새 파일 생성 금지)
  - 기존 토픽 파일이 없는 새 주제만 새 파일 생성 + 인덱스에 포인터 추가
```

**Layer 3: AIDB (벡터 검색)**
```yaml
역할: 모듈 단위 결정사항, 문제해결, 패턴
저장: 작업 성공 시 자동
검색: 작업 시작 시 키워드 기반
```

### 실행 절차

**Step 1: 현상 분석**
```yaml
수집:
  1. MEMORY.md 줄 수 확인 (200줄 한정)
  2. 모든 상세 파일 목록 + 줄 수
  3. MEMORY.md 섹션 목록 추출
  4. 상세 파일별 섹션 목록 추출
```

**Step 2: MECE 검증** (공통 원칙 1 참조)

ME (상호배타성) — 중복 점검
```yaml
검증:
  - MEMORY.md 내부 섹션 간 중복
  - MEMORY.md ↔ 상세 파일 간 중복 (포인터 제외)
  - 상세 파일 간 중복
  - 스킬과의 중복 (스킬에 있는 내용이 메모리에도 있는지)
위반_표시: (중복) [내용] — MEMORY.md L42 ↔ details.md L108
```

CE (완전망라성) — 누락 점검
```yaml
검증:
  - MEMORY.md에 포인터 없는 상세 파일 (고아 파일)
  - 상세 파일에 있지만 MEMORY.md에 요약 없는 주제
  - MEMORY.md에 포인터 있지만 상세 파일 없는 주제 (깨진 링크)
위반_표시: (누락) [내용] — 포인터/요약/파일 없음
```

분해 기준 일관성 (공통 원칙 2 참조)
```yaml
검증:
  - MEMORY.md 최상위 섹션이 동일 기준으로 분해되었는가
  - 권장 기준: 도메인별 (컴포넌트, 폼엔진, 워크셋, 인프라...)
  - 혼재 금지: 도메인(컴포넌트) + 기술(React) + 시간(2026-02)
```

**Step 3: 개선안 제시**

MEMORY.md 다이어트 (200줄 초과 시)
```yaml
이관_후보:
  1. 코드 스니펫 → 주제별 파일로
  2. 상세 규칙 (3줄+) → 주제별 파일로, MEMORY.md에는 1줄 요약
  3. 히스토리 날짜 → 제거 (주제별 파일에만 유지)
유지_대상:
  1. 실수 방지 규칙 (하지 말 것 / 할 것 패턴)
  2. 핵심 1줄 요약
  3. 포인터 링크
```

상세 파일 분할 (150줄+ 시)
```yaml
분할_기준:
  - 독립 도메인 → 별도 파일
  - 관련 주제 묶음 유지
분할_절차:
  1. 새 파일 생성 (주제별)
  2. 원본에서 해당 섹션 제거
  3. MEMORY.md 포인터 업데이트
```

**Step 4: 리포트 출력**
```markdown
# 메모리 구조 검증 리포트

## 현상
- MEMORY.md: __줄 (한도: 200)
- 상세 파일: [파일별 줄 수]

## MECE 검증
### ME (중복)
- (중복) [내용] — 위치1 ↔ 위치2
### CE (누락)
- (누락) [내용] — 이유
### 분해 기준
- [평가]

## 제안
### 즉시 (200줄 초과 해결)
1. [이관 대상] → [목적지]
### 단기 (구조 개선)
1. [분할/통합 대상]

### 정리 후 예상
- MEMORY.md: __줄 → __줄
- 파일 구조: [변경 후]

실행할까요?
```

**Step 5: 실행 (사용자 승인 후)**
```yaml
절차:
  1. 새 주제별 파일 생성 (Write)
  2. MEMORY.md에서 이관된 내용 제거 + 포인터 추가 (Edit)
  3. 원본 상세 파일에서 분할된 내용 제거 (Edit)
  4. 깨진 링크 수정
  5. 최종 줄 수 확인
원칙:
  - 내용 삭제 금지 — 이동만 허용
  - 사용자 승인 없이 실행 금지
  - 한 번에 하나의 변경 (리뷰 가능하도록)
```

### MEMORY.md 이상적 구조
```markdown
# [프로젝트명] 핵심 패턴

> 상세: [topic-a.md](topic-a.md) | [topic-b.md](topic-b.md) | ...

## 도메인A
- 규칙1 (1줄)
- 규칙2 (1줄)
- 상세 → [topic-a.md](topic-a.md)

## 도메인B
- 규칙1 (1줄)
- 상세 → [topic-b.md](topic-b.md)

(150~180줄 유지, 20줄 여유)
```

## 워크플로우 연결

```
작업 A (분해)
   ├─ 작업 B (검증)      품질 게이트: 통과 후 다음 단계
   ├─ 작업 C (폴더 변환)  실행 환경 구성
   └─ 작업 D (CSV 변환)   진행관리 스프레드시트

작업 E (메모리 정리)       독립 — 같은 MECE 원칙을 메모리 구조에 적용
```

- 표준 흐름: 분해(A) → 검증(B) → 통과 후 실행(C) + 관리(D)
- 개선 사이클: 검증(B)에서 위반/누락 발견 → 분해(A) 수정 → 재검증(B)
- 메모리 정리(E)는 분해 작업과 별개로, 메모리 파일이 비대해지거나 구조가 흐트러졌을 때 독립 실행한다.
