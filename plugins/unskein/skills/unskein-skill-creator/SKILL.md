# /unskein-skill-creator — 단계 스킬 생산 도구 (관문 ① 생산)

사용자 정의 프로세스의 **AI 단계(claim_kinds 가 비어 있지 않은 상태)가 수행할 스킬**을 만들고 점검한다. 단계 스킬은 손으로 흩어지게 쓰지 않고 이 도구를 단일 진입점으로 생산한다 — 형식 편차를 없애고(린트 통과 보장), "스킬 없는 상태(빈 칸)" 생산을 도구 수준에서 차단한다.

**이 문서가 단계 스킬 규격(전이 계약 frontmatter·린트 규칙)의 단일 출처다.**

## 0. 왜 — 스킬 1급화

AI가 밟는 모든 단계는 스킬이 있어야만 성립하고, 상태는 그 스킬 호출의 순서표다. 화면(정의 매트릭스)에서 프로세스를 만들 수 있어도, 스킬 없는 상태는 정의할 수는 있어도 실행되지 않는다. 설계 순서는 **스킬(실행 자산) → 배열(프로세스 정의) → 상태(파생)** 다.

스킬 정합성은 한 지점이 아니라 세 관문으로 보장한다(서버는 스킬 repo 를 읽을 수 없다 — 주소만 저장, 인증 비밀 없음):

| 관문 | 시점 | 무엇을 보장 | 수단 |
|---|---|---|---|
| ① 생산 | 스킬을 만들 때 | 스킬 자체 정합성 — 전이 계약 frontmatter·출력 마커·이름 slug | **이 스킬** (골격 생성 §3.2 + 린트 §3.3) |
| ② 등록 | 정의 저장·프로젝트 연결 시 | 순서·전이의 시스템 적합성 — 정의의 AI 단계마다 skill_key 존재 + 그 자리 전이 모양↔exits 일치 + doc_slot↔output 일치 | **실행기 자동 신고 기반 교차 검증**(사람의 별도 등록 절차 없음): 실행기가 설치 스킬의 폐쇄 메타를 claim 으로 자동 신고하고, 서버가 정의 저장 시 신고 계약과 대조(불일치 422)·연결 시 skill_key 완비 검증(누락 409). §3.4 매니페스트 추출은 push 전 로컬 점검용으로 유지 |
| ③ 실행 | claim·설치 시 | 실물 일치 — 설치된 스킬이 지목(skill_key)과 같은가(이름 정확 일치) | 실행기 스캔 — 설치된 것만 능력 신고, 미설치 스킬 지목 카드는 안 집음(pull — 카드는 대기), 불일치는 QUESTION 회수(fallback 금지) |

## 1. 투입 시점 = 생산 시점 (push 전)

- 이 도구는 스킬 repo 에 push 되기 **전**에 쓴다. 실행기는 클론·설치한 스킬에 이 도구를 적용해 수리하지 않는다 — **정합성이 보장된 것만 소비한다.** (설치 후 수리는 실행기마다 실물이 달라져 재현성이 무너지고, 조용한 수리는 fallback 금지 위반이다. 실행기의 몫은 신원 대조뿐, 불일치는 질문 회수.)
- **배달 = 운영자 설치**: 완성된 단계 스킬은 스킬 plugin 으로 묶여 실행기 운영자가 직접 설치한다(현행 unskein plugin 설치·업데이트와 같은 사람 게이트). 미설치 실행기는 그 프로세스 카드를 집지 않을 뿐이고 카드는 기다린다(pull 원칙 — 정상). 업데이트 = plugin 업데이트라는 의식적 행위다.

## 2. 전이 계약 — frontmatter 규격

단계 스킬 SKILL.md 의 frontmatter 필수 키 5종:

| 키 | 값 | 뜻 |
|---|---|---|
| `name` | 소문자 slug (`[a-z][a-z0-9-]*`) | 정의의 skill_key 와 **정확 일치** — 단계→스킬 선택은 이 값 하나로만 된다 |
| `description` | 자유 텍스트 | 사람용 설명(트리거 포함 권장). 선택에는 쓰이지 않는다 |
| `version` | `X.Y.Z` | 실행 대장(ledger)의 skill_version 원료. 필수 — 누락 스킬은 신고·실행 불가. 내용을 수정하면 최소 patch 를 올린다(§3.4 수정 절차) |
| `exits` | `forward` \| `verdict` \| `terminal` | 전이 모양(방법). 어느 모양이든 막히면 `QUESTION:` 회수 가능 |
| `output` | `none` \| `doc` \| `payload` | 산출물 형 |

- `exits` — **forward**: 단일 전진(수행 후 다음 status 하나로 보고) / **verdict**: PASS·FAIL 분기(판정에 따라 두 갈래 중 하나로 보고) / **terminal**: 단일 홉(수행 즉시 종결 상태로 보고).
- `output` — **none**: 산출 본문 없음(RESULT 첫 줄만) / **doc**: 문서 산출(`<<<UNSKEIN_DOC` 블록 → 정의의 doc_slot 에 저장. 슬롯 이름은 정의 소유 — 스킬은 "문서를 낸다"만 선언) / **payload**: 보고 payload 에 구조화 값.
- **구체 status 키 금지 (핵심 원칙)**: 전이 "방법"(exits 모양)은 스킬에, 전이 "대상"(구체 status 키)은 정의에 가른다. 스킬 본문·frontmatter 어디에도 특정 status 키를 박지 않는다 — 키를 박으면 그 스킬은 다른 프로세스 자리에 재배열할 수 없다. 보고할 status 값은 **작업 프롬프트(claim)가 배달한 값**만 쓴다.

## 3. 절차

### 3.1 추출 — 정의에서 스킬 요건 도출

대상 프로세스 정의(상태·전이·doc_slot·claim_kinds)와 그 단계가 다룰 작업 맥락에서 뽑는다:

- 스킬이 붙을 자리(상태)의 **전이 모양** → `exits` — 분기 자리면 verdict, 직진 자리면 forward, 접수 즉시 종결이면 terminal. 정의의 그 자리와 모양이 어긋나면 관문 ②에서 409 로 막힌다.
- 그 상태의 **doc_slot 유무** → `output` — 슬롯이 있으면 doc.
- **입력** — 이전 단계 산출물 슬롯. 스킬 본문 "입력" 절에 슬롯 이름이 아니라 **내용의 성격**으로 서술한다(예: "접수된 요청 본문").
- `claim_kinds` — 어느 실행기(kind)가 집는 단계인지(본문 맥락 서술용). 어휘의 단일 출처는 서버 토큰 kind 다(발급 시 선언 ADR-0013, `/api/whoami` 판별 ADR-0027). 단계 claim 에 유효한 값은 `mori`(EXECUTOR — WSL 헤드리스 실행기)와 `tester`(윈도우 CDP 화면검증기 — kind 는 스테이지 게이트, ADR-0014)다. `planner` 는 계획 도구 토큰이라 단계 실행기가 아니다. 추측으로 적지 않는다.
- **마감 인스펙션 제안(선택)** — 정의의 종결 상태 직전에 지식을 기록하는 단계가 없으면, §7 마감 인스펙션 단계를 넣을지 사용자에게 제안한다. 넣기로 하면 그 단계의 골격도 함께 생산한다(상태 추가는 정의 소유 — 사용자가 정의에서 한다).

정의를 조회할 수 없으면(권한·환경) 사용자에게 상태·전이·산출물을 물어 채운다. 추측으로 채우지 않는다. 정의 자체를 처음 만드는 경우면 §10 골격으로 정의 문서부터 작성한다.

### 3.2 골격 생성

§5 템플릿으로 `<스킬 repo>/…/<name>/SKILL.md` 를 생성한다. 디렉토리명 = `name` = 정의에 지목할 skill_key. "절차" 절의 검증 항목은 §8 검증 절 설계 4패턴을 참조해 채우고, 검증(TEST) 단계 스킬이면 §9 를 따른다.

### 3.3 린트 — 기계 검사

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/unskein-skill-creator/scripts/skill_lint.py" <스킬 repo 루트>
```

§4 규칙을 전부 검사한다(위반 1건 이상 = exit 1, 위반 목록 출력). 스크립트가 못 보는 것은 본문을 읽어 직접 확인한다:

- 본문 산문에 특정 프로세스의 status 키가 전이 대상으로 적혀 있지 않은가(예: "다음 status 는 `review` 다" — 재배열 불가로 만드는 반례).
- 출력 마커 예시가 §5 규약과 같은 형태인가.
- 막힘 처리(`QUESTION:` 회수)가 우회·fallback 없이 서술돼 있는가.

### 3.4 매니페스트 추출

```bash
python3 "${CLAUDE_PLUGIN_ROOT}/skills/unskein-skill-creator/scripts/skill_lint.py" <스킬 repo 루트> --manifest skills-manifest.json
```

린트 전부 통과 시에만 폐쇄 메타(`name`·`version`·`exits`·`output`)를 JSON 으로 추출한다. **스킬 본문은 서버에 올리지 않는다**(주입 경계 — 서버는 모양만 보고 적합성을 판정). 등록 API(관문 ②)가 준비되면 이 파일을 등록하고, 준비 전에는 스킬 repo 루트에 `skills-manifest.json` 으로 함께 커밋해 둔다.

**수정 절차 — 버전 올림과 재추출은 한 단위다.** SKILL.md 본문·동봉 자료를 수정하면:

1. `version` 을 최소 patch 로 올린다(예: 0.1.0 → 0.1.1).
2. 재린트(§3.3) → 매니페스트 재추출(위 명령) → 스킬과 매니페스트를 **함께 커밋**한다.

이유: 관문 ③(실행기 신원 대조)은 이름·버전으로 실물 일치를 본다 — bump 없는 수정은 "같은 버전, 다른 내용"이라는 조용한 불일치를 만들고, 재추출 없는 bump 는 낡은 매니페스트가 관문 ②에 등록되게 한다. 재추출을 잊은 낡은 매니페스트는 기본 린트가 잡는다(§4-13).

## 4. 린트 규칙 (관문 ① 기계 검사)

1. frontmatter 필수 키 5종(`name`·`description`·`version`·`exits`·`output`) 존재
2. `name` = 소문자 slug(`[a-z][a-z0-9-]*`) 이고 디렉토리명과 일치
3. **dev 예약 이름 거부** — `unskein-exec`·`unskein-verify`·`unskein-git`·`unskein-wiki-search`·`unskein-wiki-ingest`·`unskein-wiki-lint`. 디폴트 스킬(dev 프로세스)과의 충돌·선택 혼란 차단(§6)
4. 같은 repo 안 `name` 중복 거부(선택 결정성)
5. `version` = `X.Y.Z` 형식
6. `exits` ∈ {forward, verdict, terminal} / `output` ∈ {none, doc, payload}
7. 본문에 출력 마커 규약(`RESULT:` 와 `QUESTION:`) 존재
8. `RESULT:` 예시 줄의 `status=` 값은 `<…>` 플레이스홀더여야 한다 — 구체 status 키 하드코딩 거부
9. `output: doc` 이면 본문에 `<<<UNSKEIN_DOC` 블록 규약 존재
10. 스킬 디렉토리에 `CLAUDE.md` 동봉 거부 — 예약 파일(다오 상시 규약은 시스템 소유). repo 다른 위치의 CLAUDE.md 는 경고(배포 자산에 섞이지 않는지 확인)
11. UTF-8 로 읽을 수 있어야 한다
12. `RESULT:` 예시 줄에 `stage=` 가 있으면 그 값은 frontmatter `name` 과 정확히 일치해야 한다 — 실행 대장의 단계 라벨이라 복붙 잔존·플레이스홀더 미치환을 거부(status 와 달리 stage 는 생산 시점에 확정되는 값이다)
13. repo 루트에 `skills-manifest.json` 이 있으면 현재 추출본과 일치해야 한다(신선도) — 수정 후 재추출을 잊은 낡은 매니페스트 거부. `--manifest` 실행에서는 이 검사를 건너뛴다(그 실행 자체가 갱신이다)

## 5. 골격 템플릿

공통 뼈대(플레이스홀더 `<…>` 를 채운다). `exits`·`output` 값과 "보고" 절만 모양에 맞게 고른다.

````markdown
---
name: <skill-slug>
description: <이 단계가 하는 일 한 줄. 트리거 — <단계 이름>, …>
version: 0.1.0
exits: <forward|verdict|terminal>
output: <none|doc|payload>
---

# <단계 이름>

이 스킬은 프로세스 정의가 skill_key 로 지목한 단계에서, 실행기가 배달한 작업 프롬프트로 호출된다. 단계·다음 status·산출물 슬롯은 정의가 정하고 claim 이 배달한다 — 이 문서는 "무엇을 어떻게 하는지"만 담는다.

## 입력

- <이전 단계 산출물의 성격 — 예: 접수된 요청 본문. 슬롯 이름을 적지 않는다>

## 절차

1. <구체 작업>
2. <검증·확인>
3. 막히면(진전 없음, 인증·의존성 누락 등) 우회하지 말고 `QUESTION:` 으로 사유를 드러낸다(fallback 금지).

## 보고

status 값은 **작업 프롬프트가 배달한 다음 status** 를 그대로 쓴다. 이 문서에 status 키를 적지 않는다.

<보고 절 — exits 모양별로 아래에서 선택>
````

**exits: forward — 단일 전진** (output: none 이면 펜스 블록 생략)

````markdown
수행을 마치면 마지막 줄에 보고한다.

```
RESULT: status=<배달된 next status> stage=<skill-slug> summary=<한 줄 요약>
<<<UNSKEIN_DOC
<산출 본문 — output: doc 일 때만>
UNSKEIN_DOC
```
````

**exits: verdict — PASS·FAIL 분기**

````markdown
판정을 내리고, 판정에 대응하는 배달 status 로 보고한다 — PASS 면 <배달된 통과 next>, FAIL 이면 <배달된 실패 next>.

```
RESULT: status=<판정에 대응하는 배달 next> stage=<skill-slug> summary=<판정과 근거 한 줄>
<<<UNSKEIN_DOC
## 판정: PASS|FAIL
- <근거 항목>
UNSKEIN_DOC
```
````

**exits: terminal — 단일 홉 종결**

````markdown
수행 즉시 종결 상태로 보고한다(중간 상태 없음).

```
RESULT: status=<배달된 종결 status> stage=<skill-slug> summary=<한 줄 요약>
<<<UNSKEIN_DOC
<결과 본문 — output: doc 일 때만>
UNSKEIN_DOC
```
````

**막힘 — 결정 회수형 QUESTION (exits 모양과 무관한 공통 요소)**

막힘이 "사람이 골라야 풀리는" 종류(복구 방법이 여럿, 사양이냐 버그냐 판단 등)면 한 줄 질문 대신 **선택지를 담아** 회수한다 — 다오가 낙관으로 진행하는 대신 사람이 고르게 한다:

````markdown
```
QUESTION: <단계> 미완 — <대표 항목 한 줄>.
- <항목>: <근거>. 처리: <복구 선택지1> / <선택지2> / 취소
복구·선택 후 이 단계를 다시 실행해 주세요.
```
````

**QUESTION 생애주기 — 회수 이후는 시스템 소관.** QUESTION 을 회수하면 시스템이 카드를 `waiting` 으로 옮기고, 사람이 답하면(`answered`) **같은 상태에서 직전 세션을 이어받아 재개**된다. 역방향 전이·재시도 로직을 스킬에 설계해 넣지 않는다 — 스킬은 같은 상태에서 다시 호출돼도 안전하게(멱등 지향) 쓴다.

## 6. 디폴트 스킬 대체 원칙

기존 dao-skills 6개 = **디폴트 스킬이자 디폴트 단계(dev 프로세스)** 다. 사용자 스킬 plugin 이 들어와 이를 대체할 때 충돌·선택 혼란이 없어야 한다:

- **이름 충돌 금지**: dev 예약 이름(§4-3) + `CLAUDE.md`(§4-10)는 사용자 스킬이 쓸 수 없다 — 이 린트(관문 ①)와 매니페스트 등록(관문 ②) 시점에 거부된다.
- **선택 결정성**: 다오 세션 카탈로그에 같은 역할의 스킬이 둘 보여 "골라야 하는" 상황을 만들지 않는다. 단계→스킬 선택은 **claim 이 배달한 skill_key 정확 일치 하나**로만 — 다오가 비슷한 스킬을 추론으로 고르는 여지를 제거한다. description 은 사람용이고 선택에 관여하지 않는다.
- **대체 단위 = 프로세스**: dev 카드는 디폴트 스킬·단계표로, 사용자 프로세스 카드는 사용자 스킬로 — 한 카드 안에서 섞이지 않는다(트리 단일 프로세스 제약이 보장).

## 7. 선택 단계 — 마감 인스펙션 (지식 내재화)

어떤 사용자 프로세스든 **완료(종결 상태) 직전에 인스펙션 단계를 선택해 넣을 수 있다** — 작업에서 얻은 지식(패턴·교훈·결정)을 repo 에 기록해 다음 작업이 재사용하게 하는 단계다. dev 프로세스의 마감(inspect) 단계에서 `unskein-wiki-ingest`·`unskein-wiki-lint` 가 하는 역할의 사용자 프로세스 버전이다.

- **왜 완료 직전인가**: 그 시점에 작업의 산출·검증 기록이 전부 모여 있고, 아직 종결 전이라 기록을 코드와 함께 커밋에 실을 수 있다. 지식을 남기지 않는 프로세스는 작업마다 같은 재분석을 반복한다.
- **넣는 방법**: 단계(상태) 추가는 정의 소유다 — 사용자가 정의에서 종결 직전에 상태 하나를 추가하고, 이 도구는 그 자리에 붙을 스킬 골격만 생산한다(§3.1 에서 제안 → §3.2 로 생성 → §3.3 린트). 넣지 않아도 프로세스는 성립한다 — 선택이다.
- **계약 모양**: exits 는 그 자리 전이 모양을 따른다 — 종결 상태로 곧장 보고하는 자리면 `terminal`, 뒤에 단계가 더 있으면 `forward`. output 은 기록 요약을 정의의 doc_slot 에도 남기면 `doc`, repo 커밋으로만 남기면 `none`.
- **이름**: dev 예약 이름(§4-3)은 쓸 수 없다 — 프로세스 고유 slug 를 쓴다(예: `<프로세스명>-inspection`).

골격은 §5 공통 뼈대를 그대로 쓰되, "절차" 절을 아래 내용으로 채운다:

1. 작업 중 남긴 기록·산출물을 훑어 **재사용 가능한 지식만** 고른다 — 기준은 "다음 작업의 시작 비용을 줄이는가". 작업 메타데이터(상태·일정·담당)는 남기지 않는다.
2. 종류별로 repo 문서에 기록한다 — 도메인 사실·패턴·주의사항은 도메인 문서에, 기술 선택과 그 배경은 결정 기록에, 항상 적용될 핵심 규칙은 상시 문서에 한 줄 포인터로. 같은 내용을 여러 곳에 중복하지 않는다 — 이미 있으면 그 문서를 갱신한다.
3. 기록하면서 기존 문서와의 중복·충돌·낡은 내용을 함께 점검한다(문서 부패 방지).
4. 남길 지식이 없으면 지어내지 않는다 — 없다고 요약에 적고 보고한다.

## 8. 검증 절 설계 지침 — 4패턴

골격 "절차" 절의 `<검증·확인>` 을 채울 때 반복되는 설계 패턴이다(frame9-migration 단계 스킬 11종 실전에서 도출). 해당하는 상황이 있으면 처음부터 도출하지 말고 이 패턴으로 채운다. 공통 정신: **응답이 성공이라고 검증이 끝난 게 아니다 — 결과의 의미를 가르고, 숨은 실패를 드러내고, 사람 판단이 필요한 것만 QUESTION 으로 보낸다.**

1. **빈 결과의 이중 의미 분기** — "0건"은 정상(대상이 원래 없음)일 수도, 미완(선행 단계 누락·전량 매칭 실패)일 수도 있다. 무조건 QUESTION 하지 말고 두 경우를 가르는 근거(선행 산출물·원본 데이터 유무)를 확인해 분기한다. 실례: GET 파라미터 0건 = 전체조회 폼이면 정상, 파라미터가 있는데 전량 매칭 실패면 QUESTION.
2. **삼켜진 실패 드러내기** — 백엔드가 성공 응답 안에 부분 실패를 숨길 수 있다(스킵 집계만 주고 개별 실패는 로그에만). 응답 필드끼리 대조해 차이를 경고로 드러낸다. 실례: `migrated < total − skipped` 면 그 차이가 삼켜진 삽입 실패다.
3. **실패 3급 분류** — 배치 응답의 실패 항목을 곧 실패로 보지 않는다. **정상 스킵**(이미 존재) / **경고**(원본에 없는 참조 — dangling) / **차단**(DDL·스키마 오류 등 진행 불가) 으로 분류하고, 차단만 QUESTION 으로 회수한다.
4. **환경값 소비 경로 확인** — 스킬이 넘기는 환경값(조직 id 등)을 엔드포인트가 실제로 받는지 코드로 확인한다. 받는 엔드포인트가 있으면 그것을 선택하고, 안 받으면 넘기되 검증에서 경고로 드러내고, 백엔드가 다른 근거로 스스로 판정하는 값(예: bis_id 에서 파생되는 방언)은 넘기지 않는다.

## 9. 검증(TEST) 단계 생산 — verdict·tester

결과가 화면·실동작인 프로세스(폼 마이그레이션 등)는 종결 전에 검증 단계를 둔다 — `exits: verdict`, `claim_kinds: [tester]` 가 기본 모양이다. 일반 골격(§5)과 다른 점:

- **시나리오는 스킬에 박지 않는다** — 검증 시나리오·수용 기준은 카드 본문(plan_doc 류)과 이전 단계 산출물에서 받는다. 스킬 본문에는 "무엇을 받아 어떻게 검증하는지"(콘솔 에러 수집, 실제 렌더 확인, 화면 캡처 등)만 담는다. 구체 status 키 금지와 같은 원칙의 검증판 — 시나리오를 박으면 다른 프로세스 자리에 재배열할 수 없다.
- **실행 수단은 실행기 쪽이 제공한다** — CDP Chrome 접속·캡처는 실행기(TESTER)의 `unskein-test` 가, 검증 케이스(노하우) 동기는 `unskein-case` 가 담당한다. 스킬은 이 수단이 있다고 전제하고 검증 내용을 서술한다.
- **FAIL 갈래 설계 3택** — verdict 의 실패 status 를 어디로 보낼지는 정의 소유의 결정이고, 기준은 "실패 원인 단계를 아는가"다:
  1. **특정 단계 복귀** — 실패 원인이 될 단계가 하나뿐일 때(구현 단계가 하나인 프로세스). 실패 status = 그 단계.
  2. **진단 상태 신설** — 다단계 프로세스에서 실패 원인이 어느 단계인지 모를 때(기본값). 실패를 받는 진단 단계가 원인을 판정해 해당 단계로 되돌린다 — 검증 스킬에 원인 추정을 떠넘기지 않는다.
  3. **QUESTION 회수** — 실패가 "사양이냐 버그냐" 같은 사람 판단일 때. status 전이 없이 `waiting` 으로 회수한다(§5 생애주기).
- **종단 remap 은 적용되지 않는다** — PASS/FAIL 모두 `done` 으로 마감하는 종단 remap(ADR-0019)은 **CDP 검증 의뢰 카드(`payload.cdp_request`)를 tester 토큰이 보고할 때만** 발동한다. 카드 속성·토큰 kind 조건이지 프로세스 정의의 상태 조건이 아니다 — 사용자 프로세스의 verdict 단계 보고는 remap 없이 배달된 status 가 그대로 적용된다.

## 10. 프로세스 정의 문서 골격 (상태 배열)

설계 순서는 **스킬 → 배열 → 상태**(§0)인데, 배열(정의)을 처음 만드는 경우의 형식이 필요하다. 서버 등록 API(관문 ②) 준비 전에는 정의를 이 골격의 문서로 만들어 프로젝트 repo 에 둔다 — 골격대로 채우면 관문 ② 사전 검증표가 함께 채워진다.

````markdown
# <프로세스 이름> — 프로세스 정의

## 상태 배열

| 순서 | status 키 | 단계 이름 | claim_kinds | skill_key | exits | doc_slot | next (전이) |
|---|---|---|---|---|---|---|---|
| 1 | <키> | <이름> | [mori] | <skill-slug> | forward | <슬롯 또는 —> | <다음 status 키> |
| … | <키> | <검증> | [tester] | <skill-slug> | verdict | <슬롯 또는 —> | PASS→<키> / FAIL→<키> |
| n | <종결 키> | 완료 | [] | — | — | — | — (종결) |

## 프로세스 밖 요소

<프로세스가 전제하는 사전·설정(사전 데이터, 환경 준비 등) — 상태로 만들지 않는 것과 그 이유>

## 관문 ② 사전 검증

- [ ] AI 상태(claim_kinds 비어 있지 않음)마다 skill_key 가 매니페스트(skills-manifest.json)에 존재한다
- [ ] 각 자리의 전이 모양(직진/분기/종결)과 그 스킬의 exits 가 일치한다
- [ ] doc_slot 있는 상태 ↔ 스킬 output=doc 이 일치한다
- [ ] status 키가 시스템 상태(waiting·answered 등)와 겹치지 않는다
````

- **AI 단계 표기**: claim_kinds 가 비어 있지 않은 상태가 AI 단계다. 사람 단계·종결 상태는 claim_kinds `[]`, skill_key `—` 로 적는다.
- **시스템 상태 불포함**: `waiting`(QUESTION 대기)·`answered`(답변 후 재개 대기) 등 시스템이 옮기는 상태는 정의에 넣지 않는다 — 정의는 작업이 밟는 단계만 담는다.
- **등록·완료 제외**: 접수(등록)와 종결(완료) 사이의 단계만 설계 대상이다. 종결 상태는 배열의 끝점으로만 적는다.
- 관문 ② API 가 열리기 전까지는 위 검증표를 사람이 채운다. (정의 문서 ↔ 매니페스트 교차 검증의 기계화는 이 형식이 실전에서 안정된 뒤 `skill_lint.py --process` 모드로 추가할 수 있다 — 현재 미구현.)
