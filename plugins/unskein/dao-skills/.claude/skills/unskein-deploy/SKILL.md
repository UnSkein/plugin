# 배포 — 작업 마감

이 단계는 `inspect` status로 claim된다. 이전 단계(`test`)의 검증 결과를 모리가 "검증 결과"로 프롬프트에 전달한다. 검증(`unskein-verify`)을 통과한 변경과 기록을 커밋해 원격에 push 하고, 이 repo에 배포 정보가 구성돼 있으면 그대로 배포한다. 정보가 없거나 비어 있으면 임의값으로 배포하지 않고 `QUESTION:`으로 회수한다.

이 단계는 마감 직전 보조 단계 두 개를 먼저 흡수해 수행한다 (자체 status 전이 없음):

- **unskein-wiki-ingest** — `docs/raw/`의 작업 기록을 정제해 repo 지식(`docs/architecture/`·`docs/decisions/`)으로 옮기고, 처리한 원본을 `docs/raw/done/`으로 이동한다.
- **unskein-wiki-lint** — 기록한 지식 문서의 부패(깨진 링크·중복·stale)를 점검한다.

이 보조 단계의 산출물은 repo 커밋으로만 남긴다. DB 본문 컬럼에는 저장하지 않는다. 정제·점검을 마친 뒤 아래 배포 절차로 진행한다.

## 1. 배포 정보 확인 — 먼저 한다

배포에 필요한 정보가 repo에 구성돼 있는지 먼저 본다. 다음을 찾는다 (있는 것만):

- repo의 배포 설정 문서·파일 (예: `docs/deploy.md`, `.unskein/deploy.yml`, `CLAUDE.md`의 배포 단락 등).
- 필요한 항목: **대상 원격·브랜치**, **서버 재배포 방법·접근**(backend가 있을 때), **빌드 명령**(frontend가 있을 때), **마이그레이션 방식**(DB가 있을 때 — 자동/수동).

판단:

- 필요한 정보가 다 구성돼 있으면 → 2번으로 진행한다.
- 없거나 비어 있으면 → 바로 **회수**한다 (이 스킬 마지막 단락). 임의값·추측으로 배포하지 않는다 (fallback 금지).

## 2. 커밋

1. 검증을 통과한 변경과 위키 기록을 함께 커밋한다 (코드와 지식이 한 번에 push 되도록).
2. 작업과 무관한 변경(예: `.gitignore`)이 섞여 있으면 pathspec으로 분리해 별도로 다루거나 빼고, 받은 작업에 직접 닿는 변경만 커밋한다.
3. 비밀(토큰·키)이 변경·커밋·로그·원격 주소에 남지 않는지 확인한다. 남았으면 멈추고 회수한다.

## 3. 머지

- 작업 브랜치를 배포 대상 브랜치(배포 정보의 값)에 머지한다. 이력을 남기려면 `--no-ff`를 쓴다.
- 충돌이 나면 임의로 한쪽을 버리지 말고, 해결 근거가 불명확하면 회수한다.

## 4. push

- 원격에 push 한다 (인증은 클라이언트가 갖춘 git 자격증명을 따른다 — 토큰을 원격 주소에 넣지 않는다).
- push 인증이 누락·실패하면 우회하지 말고 회수한다 (fallback 금지).

## 5. 서버 재배포 (backend 변경이 있을 때)

- 배포 정보의 재배포 방법을 따른다: frontend 빌드 + backend 재시작 등.
- CI/CD가 없으면 이 단계는 수동임을 전제로 한다 — push 만으로 서버에 반영되지 않는다.
- 재배포 접근(서버 주소·실행 방법)이 배포 정보에 없으면 회수한다.

## 6. 마이그레이션 (DB 변경이 있을 때)

- 방식은 배포 정보에서 읽는다.
  - 자동(예: 시작 훅에서 실행)이면 재시작만으로 적용된다 — 별도 명령을 돌리지 않는다.
  - 수동이면 배포 정보가 지정한 마이그레이션 명령을 실행한다.
- 자동/수동 여부가 배포 정보에 없으면 추측하지 말고 회수한다.

## 7. 미러·유출 점검

- 두 repo에 byte-identical로 유지해야 하는 파일(예: orchestrator 미러)이 배포 정보에 적혀 있으면, 양쪽이 동일한지 점검하고 어긋나면 동기화한다.
- 심링크·`.gitignore`로 막아야 할 유출(예: dogfooding 심링크가 배포 repo에 실리는 것)이 없는지 점검한다. 새는 것이 있으면 커밋에서 뺀다.

## 회수 (QUESTION) — 이 스킬의 핵심

배포에 필요한 정보가 없거나 비어 있으면, 임의값으로 배포하지 말고 `QUESTION:`으로 회수한다.

- **무엇이 없는지 구체적으로 묻는다**: 대상 원격·브랜치 / 서버 재배포 방법·접근 / 빌드 명령 / 마이그레이션 방식 중 빠진 항목을 짚는다.
- 사람이 답하면, 그 답을 **repo의 배포 설정에 기록**한 뒤(다음 배포가 재사용하도록) 같은 세션에서 배포를 이어간다.
- 절대 추측·기본값으로 배포하지 않는다 (fallback 금지). 정보가 확정돼야 배포한다.

배포 스킬의 존재와 필요한 배포 정보는 모리 클라이언트 셋업(`unskein-connect`·`unskein-add-site`) 단계에서 안내·수집하는 것을 전제로 한다. 셋업에서 수집되지 않은 항목은 첫 배포 때 이 회수로 메운다.

## 보고 — 종결 본문을 산출 본문으로 낸다

배포까지 마치면 변경 요약·배포 결과를 `RESULT` 마커의 `<<<UNSKEIN_DOC` 블록에 담아 보고한다. 모리가 이 본문을 `close_doc`에 저장한다 (done 카드 표시). 보고 next status는 `done`이다.

```
RESULT: status=done stage=deploy summary=<한 줄 요약>
<<<UNSKEIN_DOC
## 변경 요약
- <커밋·머지·push 결과>
## 배포 결과
- <서버 재배포·마이그레이션 결과 (해당 시)>
UNSKEIN_DOC
```

배포 정보가 없거나 push 인증이 누락·실패하는 등 막히면 `RESULT` 대신 `QUESTION:`으로 회수한다.
